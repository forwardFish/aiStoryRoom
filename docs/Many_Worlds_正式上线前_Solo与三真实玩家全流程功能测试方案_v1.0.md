## 1. 测试原则

### 1.1 真人操作定义

允许：

- 可见浏览器中点击、键盘输入、滚动、刷新、前进后退和打开邀请链接；
- 浏览器开发者工具或自动化框架只读采集截图、录像、控制台和网络日志；
- 测试结束后通过受控 API/数据库只读查询核对状态；
- 官方支付沙箱页面中的真实点击。

不允许：

- 用 `page.evaluate(fetch(...))` 或脚本直接提交关键业务 API；
- 直接注入 Bearer token、cookie、localStorage、runId 或角色归属；
- 在一份浏览器存储中轮流伪装三名玩家；
- 直接写数据库修复房间或额度后继续并宣布通过；
- 用 mock DOM、隐藏按钮调用、坐标猜测或跳过不可点击步骤；
- 用规则 AI 的结果冒充真实 AI 提供商验证；
- 用 Run11/Run12 或其他提交的截图替代本次新运行。

### 1.2 三名玩家隔离要求

必须使用三个独立浏览器来源：

- Origin One：独立用户数据目录，玩家 A，房主；
- Origin Two：独立用户数据目录，玩家 B；
- Origin Three：独立用户数据目录，玩家 C。

三者必须有独立 cookie/localStorage/sessionStorage/cache。可使用三个浏览器配置文件、三个无痕上下文或三台设备；单纯三个标签页不合格。

### 1.3 失败即停规则

遇到以下问题立即把当前 attempt 标为 FAIL，保存现场，再修复并启动全新 attempt：

- 页面白屏、无限加载、关键按钮不可见/不可点击；
- 401/403 后未正确回到登录或原页面；
- 5xx、未处理 Promise、明显控制台错误；
- 回合状态在三名玩家之间不一致；
- 重复扣额度、重复订单、无法继续回合；
- 玩家看见他人秘密、内部结局键或调试数据；
- 需要人工改数据库、直接调接口或刷新多次“碰运气”才能继续；
- 证据采集缺失，无法证明关键断言。

修复后禁止沿用原 runId 宣布通过；必须建立新 runId 并从 S00/M00 重跑。

## 2. 环境、账号与数据准备

### 2.1 发布候选环境

开始前记录：

| 字段 | 必填内容 |
|---|---|

### 2.2 账号矩阵

| Player-B | Multiplayer 成员 | 已验证 | 足够解锁 |

### 2.3 固定游戏数据

- 主验收世界：桑田诏。
- Solo：选择一个明确角色，记录角色 ID 和显示名；另 5 个角色由 AI 驱动。
- Multiplayer：A/B/C 选择三个不同角色，剩余角色由 AI 驱动。
- 七轮必须全部完成；不能用管理接口跳到最后一轮。
- 每轮玩家行动应预先定义一条可识别但不包含隐私的输入，用于检查是否串人、丢失或重复。
- 若 Caesar 页面仍显示可玩，必须另建 Caesar attempt，重复全部 Solo 和 Multiplayer 主流程。

| ID | 真人操作 | 页面期望 | 必须核对 |
|---|---|---|---|
| S07 | 选择 Solo 和目标角色 | 角色信息清楚 | 角色显示名、目标和隐私说明正确 |

创建动作网络断开、快速双击和服务端 409 必须有明确反馈且不能产生重复 run。

### 5.3 七轮操作

每一轮均执行以下循环，不能只保存第 1 和第 7 轮：

1. 记录轮次、阶段、公开状态、个人状态和可选动作。
2. 在可见输入控件中填写本轮主行动并点击提交。
3. 确认按钮 loading、防双击、提交成功提示和不可二次修改规则。
4. 如本轮支持谋略行动，从页面选择并提交；如不支持，UI 应准确说明。
5. 如有反应阶段，从页面选择反应；无反应时选择页面提供的跳过。
6. 等待 AI 角色完成；页面显示有限等待状态，不得无限转圈。
7. 进入下一轮，核对上一轮摘要、个人影响和新阶段。
8. 保存截图、关键请求和只读状态回读。

| ID | 轮次 | 附加断言 |
|---|---:|---|
| S12 | 第 1 轮 | 首次操作引导、按钮状态、AI 首次响应 |
| S13 | 第 2 轮 | 刷新恢复；重复提交返回幂等/冲突而非重复执行 |
| S14 | 第 3 轮 | 解锁门槛出现的时机和文案准确 |
| S15 | 解锁 | 余额足够时一次扣除；不足时进入 S16 |
| S16 | 购买回跳 | 官方沙箱支付成功后回到同一 run/round |
| S17 | 第 4-6 轮 | 解锁保持、AI 超时/重试不重复推进 |
| S18 | 第 7 轮 | 完成后只能进入结果状态，不出现第 8 轮 |

AI 故障分支至少一次：制造超时或可控 5xx，确认 UI 提示、重试上限、规则降级和回合幂等；不得在真实主证据里手工改状态跳过。

### 6.3 七轮逐轮操作矩阵

每轮必须对三个 origin 分别保存“提交前、提交后、阶段结算后”证据。建议输入可识别文本：`A-R1-main`、`B-R1-main`、`C-R1-main`，但玩家界面不应把这些测试标签展示给无权查看的人。

| 轮次 | A 操作 | B 操作 | C 操作 | 核心断言 |
|---:|---|---|---|---|
| 1 | 主行动 + 可选谋略 | 主行动 | 主行动 + 可选反应 | 顺序与提交状态正确；秘密不串人 |
| 2 | 主行动 | 主行动 + 谋略 | 主行动 | 刷新一方，状态可恢复；其他人不中断 |
| 3 | 主行动 | 主行动 | 主行动 | 解锁门槛在正确时机出现，未解锁方不能越过 |
| 4 | 解锁后主行动 | 解锁后主行动 | 沙箱购买后回跳并提交 | 三方各自权益独立，房间同步不被支付回跳破坏 |
| 5 | 主行动 | 主行动 | 主行动 | 执行非房主掉线 M21-M23 |
| 6 | 主行动 | 主行动 | 主行动 | 执行房主掉线/恢复 M24-M27 |
| 7 | 主行动 | 主行动 | 主行动 | 只完成一次；三方进入各自结果页 |

每轮结算后检查：

- 三方 `round/stage/status` 一致；
- 已提交者不可重复执行，未提交者仍可正常操作；
- 公开事件在三方一致，个人叙事只对对应玩家可见；
- AI 角色行动数量正确且不重复；
- 网络重试不会重复推进或重复扣额度；
- 没有隐藏内部键、服务器堆栈、原始 prompt 或其他玩家输入泄漏。

### 6.4 掉线、接管与恢复

| ID | 场景 | 真人操作 | PASS 断言 |
|---|---|---|---|

| M22 | AI 接管 | 等待产品定义的超时 | 仅接管 B，对应动作只执行一次 |

## 8. API 与数据库只读回读

UI 操作完成后才能执行回读。回读用于核实，不得代替 UI 操作。

### 8.1 Solo 回读

- run 的 world、mode、status、round/stage、human role、AI roles；
- 七轮行动总数和每轮幂等键；
- 解锁记录、账本记录、支付订单关联；
- 结果投影、分享快照、撤销状态；
- account history 与 run 关联。

### 8.2 Multiplayer 回读

- 单一 room/run，三名用户和三个唯一角色；
- 准备/开始时间、七轮阶段迁移、AI 接管与认领事件；
- 每个动作的所有者、轮次、状态和幂等键；
- 三方解锁权益互不串账；
- 同一公共结局与三份合法个人投影；
- 分享记录所有者和访问状态。

### 8.3 一致性断言

1. UI 显示 round/stage 与 API/DB 完全一致。
2. 每个用户可见的结果字段是服务端允许字段的子集。
3. 订单金额、账本增减、聚合余额和解锁消耗可守恒。
4. 同一业务幂等键至多产生一项有效副作用。
5. completed run 不再接受新行动或再次结算。
6. 被撤销的分享 token 不再返回快照。

所有查询语句或只读工具命令保存到 `06-api-readback`、`07-db-readback`，敏感字段只保留哈希或遮蔽值。

## 9. 安全、错误与边界矩阵

| 类别 | 必测场景 | 期望 |
|---|---|---|
| 未认证 | 直接进入 game/result/account/admin | 登录回跳或 401/403，不泄漏数据 |
| 越权 | Outsider 读取他人 run/room/result/refund | 403/404，响应不区分可枚举秘密 |
| 输入 | 空值、超长、特殊字符、HTML/脚本 | 客户端/服务端双重校验，安全渲染 |
| 并发 | 双击创建、双击提交、多人同时最后提交 | 单一副作用，阶段只推进一次 |
| 会话 | 过期、退出、多标签页、账号切换 | 不串账号，原意图可恢复 |
| 网络 | 离线、慢网、超时、请求重试 | 明确状态，可安全重试，不丢局 |
| AI | 429、5xx、超时、无效 JSON | 有限重试/降级，回合不重复推进 |
| 支付 | 伪签名、重复/乱序 Webhook | 拒绝或幂等，账本守恒 |
| 隐私 | 跨角色秘密、内部 key、日志 | 不可见、不返回、不记录敏感正文 |
| 路由 | 深链刷新、前进后退、无效参数 | 唯一正式页面壳和友好错误 |
| 兼容 | 桌面 Chrome/Edge、目标移动视口 | 核心流程可见、可点、可读 |

任何安全或账本不一致缺陷不得降级为“上线后再修”。

## 10. 生产部署后最小冒烟

生产冒烟不能重跑高风险破坏性测试，只执行最小安全链路：

1. 打开首页、世界列表、桑田诏详情、认证页、账户页、结果页无效参数分支。
2. 使用专用生产冒烟账号登录并退出。
3. 创建一局标记为 smoke 的 Solo，完成到第 1 轮提交；按产品支持方式安全终止或保留审计。
4. 创建一个私有 Multiplayer 房间，第二个冒烟账号通过邀请加入，确认后不启动或按支持方式清理。
5. 创建一条无敏感内容的分享并撤销。
6. 只读核对健康、版本、数据库连接、邮件和监控事件。
7. 支付只检查配置/签名端点健康；除非发布负责人明确批准，不在生产产生订单。

冒烟证据必须记录精确 Vercel deployment、Railway deployment/API version 和候选 SHA。

## 11. 总体判定规则

### PASS

同时满足：

- B00-B08、S00-S27、M00-M32 以及适用的 P00-P12 全部通过；
- Solo 和 Multiplayer 都是冻结候选上的全新 run；
- 三真人来自三个独立浏览器来源；
- 七轮逐轮证据、网络日志、控制台日志、API/DB 回读齐全；
- 内部键与敏感信息扫描 0 个不可接受命中；
- Web/API/部署/迁移版本一致；
- 总清单校验通过且 verdict 为 PASS。

### FAIL

出现任一业务、隐私、安全、支付、数据一致性或证据完整性失败；包括“修复后只补拍一张图”但没有从头新跑。

### BLOCKED_BY_ENVIRONMENT

仅用于浏览器控制、邮件沙箱、支付沙箱、部署平台或测试数据库客观不可用，且已保存错误原文和排查记录的情况。它不是 PASS；发布结论仍为 NO-GO。

### REPAIR 规则

每个失败建立最小修复任务：

1. 记录复现步骤、请求 ID、runId、截图和日志；
2. 定位到单一契约/模块；
3. 增加能先失败的回归测试；
4. 完成修复并运行模块测试；
5. 创建新的 attemptId，从受影响主流程起点完整重跑；
6. 不覆盖失败证据，保留前后对照。
