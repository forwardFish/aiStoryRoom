# Many Worlds｜AI 剧情连续性与动态推演引擎完整方案 v1.0

> 适用项目：`aiStoryRoom`  
> 首个落地剧本：《桑田诏：嘉靖财政危局》  
> 文档目标：解决“玩家作出选择后只得到孤立结果、下一决策与上一段剧情缺少桥接、人物反应像模板、长期因果无法自然回收”的核心问题。  
> 核心结论：**剧本负责边界与必达冲突，规则引擎负责事实和数值，AI 负责在事实边界内组织最符合当前局势的下一段故事。**

---

## 1. 当前问题与根因

### 1.1 玩家看到的问题

当前体验容易出现以下断裂：

1. 玩家提交决策后，结果只解释“选择产生了什么影响”，没有继续讲“事情如何发生”；
2. 结果故事结束后立即出现下一组选项，中间缺少人物行动、场景变化和新压力；
3. “你的判断、各方动向、局势走向”等信息虽然完整，但仍像分析报告，不像连续故事；
4. 主动谋划能修改状态，却没有自然进入主线剧情；
5. 两三天前的选择只有数值变化，没有在后续人物言行中被重新提起；
6. AI 每次只处理当前输入，容易忘记人物上一次的态度、已知信息、承诺和冲突；
7. 内部键、状态名或模板文本可能直接进入玩家剧情；
8. 剧情的场景、时间、视角和人物位置缺少统一约束，容易跳场或自相矛盾。

### 1.2 当前工程已经具备的基础

本项目并不是从零开始，已有能力包括：

- 《桑田诏》七天主线、十二个主线决策、角色设定、结局和每日压力；
- `StoryRun.version` 乐观锁和运行快照；
- `StoryEvent`、决策历史、证据账本、责任账本、关系与 `FateSeed`；
- 决策状态补丁、角色反应、因果回响和日终总结；
- DeepSeek 导演调用和结构化结果规范化；
- 前端流式文字、叙事页、决策页和关键事件弹窗；
- 主线决策、主动谋划、关键事件和最终裁决。

相关现状见：

- `apps/api/src/mvp-causal-runtime.ts:892`：AI 输出会覆盖规则 fallback 中的叙事字段；
- `apps/api/src/mvp-causal-runtime.ts:936`：已有结构化输出校验；
- `apps/api/src/mvp-causal-runtime.ts:1057`：已有 FateSeed 条件触发；
- `apps/api/src/mvp-causal-runtime.ts:1099`：日终总结主要由规则模板生成；
- `apps/api/src/story.service.ts:441`：节点导演目前在 API 请求中同步执行；
- `apps/web/public/app.js:271`：前端已有结果流式展示；
- `apps/web/public/app.js:666`：前端目前负责把多个结果字段拼成一段故事。

### 1.3 真正缺失的能力

当前缺少的不是“再调用一次大模型”，而是以下五个中间层：

```text
连续性上下文构建器
剧情节拍规划器
角色认知与动机模拟器
叙事一致性校验器
统一 NarrativeEntry 持久化层
```

目前规则、状态、角色反应和文案已经存在，但它们没有被组织成一条统一的“场景之前发生了什么 → 玩家做了什么 → 谁如何反应 → 局势变成什么 → 下一刻为什么需要决策”的链路。

## 2. 设计目标与非目标

### 2.1 设计目标

每次玩家决策或谋划后，系统必须生成一段符合以下结构的连续剧情：

```text
承接上一场景
→ 明确玩家行动如何被执行
→ 展示最先受到影响的人
→ 展示一项可感知的局势变化
→ 回收或埋下一条因果线索
→ 产生下一项具体压力
→ 压力自然导出下一次决策
```

玩家读完后应能回答：

1. 我现在在哪里、是什么时间；
2. 我刚做了什么；
3. 谁因此改变了态度或行动；
4. 当前最危险的问题是什么；
5. 为什么现在要做下一次选择。

### 2.2 非目标

本方案不追求：

- 让 AI 完全自由改写《桑田诏》的历史背景；
- 让 AI 自由增加天数、主线决策数量或新结局；
- 用随机成功率决定剧情；
- 让角色知道其视角之外的秘密；
- 每次把完整剧本和全部历史记录发给模型；
- 用大量数值和技术日志代替故事；
- 让前端继续承担剧情拼接和业务判断。

## 3. 核心设计原则

### 3.1 双轨叙事：固定骨架，动态过程

《桑田诏》的七天主题是必达骨架：

| 天数 | 必达冲突 | 可以动态变化的内容 |
|---|---|---|
| 第 1 天 | 改桑令、巡抚请命、商会递帖 | 谁先取得主动、总督是否公开表态 |
| 第 2 天 | 县令密信、地方催政 | 密信可信度、巡抚压力来源、证据去向 |
| 第 3 天 | 粮价上涨、巡抚急奏、商会控粮 | 粮价责任、奏报口径、官商关系 |
| 第 4 天 | 暗账浮出、巡抚察觉 | 暗账完整度、谁掌握副本、是否灭证 |
| 第 5 天 | 互相弹劾、内阁与司礼监介入 | 弹劾对象、责任叙事、内廷态度 |
| 第 6 天 | 京师回批、最终奏报 | 谁被写入奏报、证据是否成链、最终盟友 |
| 第 7 天 | 御前裁决 | 根据事实、责任、人物关系和长期因果裁决 |

AI 可以改变“过程、人物反应、证据路径和责任解释”，不能跳过必达冲突。

### 3.2 状态先于文案

系统先计算权威事实，再写故事：

```text
玩家行动
→ ActionGuard
→ 规则结果与状态补丁
→ 角色认知更新
→ 事件与因果线程更新
→ AI 剧情规划
→ AI 文案生成
→ 一致性校验
→ 持久化与展示
```

AI 不直接决定核心数值，也不能在文案中发明没有落账的事实。

### 3.3 先规划，后写作

单次大 Prompt 同时做规则判断、角色模拟和小说写作，很容易顾此失彼。应拆成两个阶段：

1. `NarrativePlanner`：只输出结构化剧情计划；
2. `NarrativeWriter`：只把已批准的计划写成角色视角故事。

规划结果先通过规则校验，再允许生成正文。

### 3.4 每段故事只推进一个主要矛盾

每次结果故事最多包含：

- 1 个主要结果；
- 1—2 个角色反应；
- 1 项世界变化；
- 1 条回收或新增线索；
- 1 个下一场景钩子。

避免一段文字同时讲完所有角色和全部变量。

### 3.5 玩家视角优先

玩家只能看到浙江总督能够感知、收到或合理推断的信息。

错误：

> 巡抚心中决定明日越级弹劾你。

正确：

> 巡抚离府时没有走正门。半个时辰后，驿站有人看见他的亲信换了快马。

## 5. 连续性上下文模型

### 5.1 六层上下文

每次推演只组装与当前场景相关的六层信息。

#### A. World Bible：不可改变的世界规则

包含：

- 嘉靖三十五年的时代约束；
- 官职权限与交通通信速度；
- 改稻为桑、国库、粮价和地方治理背景；
- 禁止现代术语、现代制度和越权行为；
- 七天主题和最终裁决边界。

来源：`docs/10_桑田诏_完整故事局剧本_v1.3_持续叙事流修订版.md` 及未来 World Package。

#### B. Run Canon：本局已经确认的事实

例如：

```json
{
  "factId": "fact_day1_audit_first",
  "subject": "zhejiang_governor",
  "predicate": "ordered",
  "object": "audit_land_register_before_reform",
  "sourceEventId": "evt_decision_001",
  "day": 1,
  "visibility": ["zhejiang_governor", "xunfu"],
  "confidence": "confirmed"
}
```

已确认事实不可被后续 AI 否定，只能被补充、重新解释或证明为角色当时的误判。

#### C. Scene State：当前场景锚点

每个场景必须有：

```json
{
  "day": 2,
  "time": "午后",
  "location": "杭州总督府偏厅",
  "povRoleKey": "zhejiang_governor",
  "presentRoleKeys": ["zhejiang_governor", "county_magistrate"],
  "sceneGoal": "判断县令密信是否可信",
  "activePressure": "巡抚要求当日答复",
  "entryFactIds": ["fact_..."],
  "openQuestion": "田亩名册的底稿在谁手中"
}
```

下一段正文必须从该锚点承接，不能无解释跳到新地点或新时间。

#### D. Character Minds：角色认知与动机

每个主要角色维护：

```text
公开目标
真实目标
当前恐惧
当前欲望
对玩家态度
已知事实
未知事实
错误认知
已作承诺
上一行动
下一倾向
```

角色反应由“目标 + 已知事实 + 当前压力 + 与玩家关系”决定，不由 AI 随意选择。

#### E. Story Threads：未完成剧情线程

每条线程维护：

```json
{
  "threadId": "thread_land_register_origin",
  "title": "田亩名册底稿来源",
  "status": "open",
  "openedAtDay": 1,
  "lastTouchedAtDay": 2,
  "importance": 90,
  "relatedFactIds": ["fact_..."],
  "relatedRoleKeys": ["county_magistrate", "xunfu"],
  "dueByDay": 4,
  "possiblePayoffs": ["evidence", "betrayal", "exoneration"]
}
```

线程超过两天未触碰时，Context Builder 应提高其召回权重；到达 `dueByDay` 必须回收、升级或明确延期。

#### F. Recent Narrative Window：最近正文

只保留：

- 当前场景上一段全文；
- 最近 3—5 个 NarrativeEntry 摘要；
- 最近一次玩家决策原文；
- 最近一次角色直接对话；
- 最近一次场景结束钩子。

这保证语言和动作承接，而不是把整局历史全部塞入上下文。

### 5.2 上下文优先级

推荐 Token 分配：

| 内容 | 建议预算 | 截断规则 |
|---|---:|---|
| 系统规则与输出 Schema | 1,500—2,000 | 不截断 |
| World Bible 当前日片段 | 1,000—1,500 | 仅当前日与相邻日 |
| Scene State | 500—800 | 不截断 |
| 当前相关角色 Minds | 1,500—2,500 | 最多 3 个角色 |
| Run Canon 检索结果 | 1,500—2,500 | 相关度排序 |
| Story Threads | 800—1,200 | 最多 8 条 |
| Recent Narrative | 2,000—4,000 | 最近优先 |
| 玩家行动与规则结果 | 500—1,000 | 不截断 |

## 6. 决策如何变成最符合当前局势的剧情

### 6.1 玩家行动解释

预设选项直接带有策划标签；自定义行动先提取结构：

```json
{
  "intent": "拖延改桑并核查田亩",
  "method": "要求巡抚三日后呈报详细田亩册",
  "targetRoleKeys": ["xunfu"],
  "usedResources": ["governor_authority"],
  "usedLeverageKeys": [],
  "riskPosture": "cautious",
  "publicness": "public_order",
  "expectedEffect": "delay_and_audit"
}
```

ActionGuard 只判断合法性；Narrative Planner 判断行动在当前局势中如何发生。

### 6.2 规则结果

规则层生成权威结果：

```json
{
  "outcome": "partial_success",
  "statePatch": {
    "暗账完整度": 8,
    "巡抚敌意": 5,
    "县令信任": 4
  },
  "createdFacts": ["fact_audit_ordered"],
  "openedThreads": ["thread_register_source"],
  "affectedRoleKeys": ["xunfu", "county_magistrate"],
  "requiredNextPressure": "巡抚要求限定核查期限"
}
```

### 6.3 剧情节拍规划

Planner 输出：

```json
{
  "sceneBridge": "承接巡抚在偏厅请命的场景",
  "beats": [
    {
      "type": "player_action_execution",
      "factIds": ["fact_audit_ordered"],
      "description": "总督要求先核田亩，并限定巡抚三日呈册"
    },
    {
      "type": "role_reaction",
      "roleKey": "xunfu",
      "description": "巡抚表面领命，实际要求把期限写入公文"
    },
    {
      "type": "world_consequence",
      "description": "县令得到喘息，但改桑进度暂缓"
    },
    {
      "type": "thread_hook",
      "threadId": "thread_register_source",
      "description": "名册底稿的来源开始成为新疑点"
    }
  ],
  "nextPromptReason": "商会得知核田后，提出先垫粮银换取保护",
  "endHook": "门外传来商会会首已候了半个时辰"
}
```

### 6.4 选择下一剧情发展的评分

如果多个剧情走向都合法，按以下权重选择：

```text
当前冲突相关度          30%
玩家行动因果匹配度      25%
角色动机一致性          20%
未完成线程回收价值      15%
当日节奏与下一钩子      10%
```

禁止用纯随机决定主线结果。可以在同分的语言细节和次要角色动作中使用低幅度随机性。

## 7. AI 两阶段输出协议

### 7.1 NarrativePlanner 输入

```json
{
  "worldRules": {},
  "dayArc": {},
  "sceneState": {},
  "playerAction": {},
  "authoritativeOutcome": {},
  "relevantFacts": [],
  "characterMinds": [],
  "openThreads": [],
  "recentNarrative": [],
  "requiredNextPrompt": {}
}
```

### 7.2 NarrativePlanner 输出

必须是 JSON，至少包含：

```text
sceneBridge
beats[]
factIdsUsed[]
roleKnowledgeUsed[]
threadsTouched[]
newSceneState
nextPromptReason
endHook
continuityClaims[]
```

Planner 不输出最终正文，不允许修改 `statePatch`。

### 7.3 NarrativeWriter 输入

Writer 只接收：

- 已批准的剧情计划；
- 当前视角与文风；
- 最近一段正文；
- 必须使用的中文名称字典；
- 字数、段落和禁用词约束。

### 7.4 NarrativeWriter 输出

```json
{
  "title": "先核田亩",
  "paragraphs": [
    "你没有立刻接过巡抚递来的名册……",
    "巡抚沉默片刻，拱手领命……",
    "他退到门边时，幕僚从廊下递来一张商会名帖……"
  ],
  "endHook": "商会会首已经在外候了半个时辰。",
  "visibleFactIds": ["fact_audit_ordered"],
  "nextPromptEventId": "evt_day1_decision2"
}
```

建议长度：

- 普通决策结果：350—700 中文字；
- 主动谋划结果：180—400 中文字；
- 关键事件结果：300—600 中文字；
- 日终故事：250—450 中文字；
- 最终裁决：800—1,500 中文字。

## 8. 连贯性校验器

### 8.1 硬校验

以下任一失败都不能发布正文：

1. 文案引用了不存在的 `factId`；
2. 文案声称的结果与 `statePatch` 冲突；
3. 角色使用了其 `knownFacts` 中没有的信息；
4. 时间倒退或无解释跳日；
5. 地点或在场人物与 Scene State 冲突；
6. 角色已经离场却直接发言；
7. 已使用筹码再次被当作可用筹码；
8. 出现内部键，如 `county_letter`、`xunfu`、`decision_1`；
9. AI 修改固定日程、决策数量或结局规则；
10. 下一决策与 `nextPromptReason` 没有语义关联。

### 8.2 软校验

按 0—100 评分：

| 维度 | 最低分 |
|---|---:|
| 上下文承接 | 85 |
| 玩家行动体现 | 90 |
| 角色动机一致 | 85 |
| 视角正确 | 95 |
| 下一钩子自然 | 80 |
| 文风符合《桑田诏》 | 75 |

总分低于 85 时进入一次 Writer repair；硬校验失败时回到 Planner repair。最多修复 2 次，仍失败则使用规则模板 fallback，并记录 `narrative_quality_degraded`。

### 8.3 名称字典与输出净化

所有内部键必须经过 `LabelDictionary`：

```json
{
  "county_letter": "清流县令密信",
  "land_contract_fragment": "田契暗账半页",
  "coastal_report": "海防军报",
  "county_magistrate": "清流县令",
  "xunfu": "浙江巡抚"
}
```

最终输出执行内部键扫描：

```regex
\b[a-z][a-z0-9]*(?:_[a-z0-9]+)+\b
```

命中后禁止进入公共 NarrativeEntry。

## 9. 统一叙事数据模型

### 9.1 NarrativeEntry

建议新增权威叙事条目，而不是由前端拼接 `message + causalCard + roleAction`：

```text
id
runId
sequence
day
sceneId
entryType
title
body
povRoleKey
sourceEventIds
visibleFactIds
threadIds
nextPromptEventId
status
createdAt
```

`entryType` 限定为：

```text
opening
story
decision_result
maneuver_result
role_impact
critical_event
day_end
final_judgement
```

### 9.2 CanonFact

```text
id, runId, subject, predicate, objectJson,
sourceEventId, day, confidence, visibilityJson,
supersedesFactId, createdAt
```

事实默认不可删除。角色误判使用 `confidence=believed`，后续可由新事实纠正，不直接覆盖历史。

### 9.3 CharacterMind

```text
runId, roleKey, day, knownFactIds, believedFactIds,
currentFear, currentDesire, attitudeJson,
commitmentsJson, lastAction, nextInclination, version
```

### 9.4 StoryThread

```text
id, runId, title, status, importance,
openedAtDay, lastTouchedAtDay, dueByDay,
relatedFactIds, relatedRoleKeys, possiblePayoffsJson
```

### 9.5 SceneSnapshot

```text
id, runId, day, time, location, povRoleKey,
presentRoleKeys, sceneGoal, activePressure,
openQuestion, entryFactIds, createdAt
```

### 9.6 StoryEvent 顺序

现有 `StoryEvent` 应增加单局单调递增的 `sequence`，并建立：

```text
@@unique([runId, sequence])
@@index([runId, day, sequence])
```

NarrativeEntry 和状态投影都以 sequence 排序，避免只依赖时间戳造成重放顺序不确定。

## 10. 前端体验与三类页面

结合当前产品决策，主游戏只保留三类视觉模式：

### 10.1 文字说明页

用于：

- 前情介绍；
- 决策结果；
- 主动谋划结果；
- 他人影响；
- 日终故事；
- 最终裁决。

行为：文字流式展示，全文完成后显示“继续”。

### 10.2 决策页

用于主线决策和关键事件回应。正文不与决策组件同时堆叠；玩家点击“继续”后切换到同背景的决策页。

### 10.3 紧急事件弹窗

只有必须立即引起注意的关键事件使用弹窗。普通角色反应、数值变化和谋划结果不得弹窗。

### 10.4 严格释放顺序

```text
结果 NarrativeEntry 已持久化
→ 前端开始流式显示
→ 全文显示完成
→ 显示“继续”
→ 玩家点击继续
→ activePrompt 决策页可见
```

后端可以提前生成下一 Prompt，但公共投影必须使用 `revealAfterEntryId` 控制，不能提前显示。

## 11. 《桑田诏》专用连续性规则

### 11.1 五条长期主线

整局持续追踪：

```text
改桑执行线
粮价与民生线
暗账证据线
督抚权力线
御前责任解释线
```

每次主线决策必须至少触碰一条，最多触碰三条。

### 11.2 角色反应优先级

```text
直接受到玩家行动影响的角色
> 当前日主线角色
> 掌握相关证据的角色
> 变量阈值触发角色
> 其他角色
```

每次最多生成两个可见角色反应，避免所有角色轮流发言。

### 11.3 跨日回收

- 第 1 天的公开命令应在第 2—3 天以公文、口碑或巡抚态度回响；
- 第 2 天的密信处理应在第 4 天暗账浮出时重新定性；
- 第 3 天的粮价与商会选择应在第 5 天弹劾中成为责任证据；
- 第 4 天暗账去向必须进入第 6 天最终奏报；
- 第 5 天对内阁和司礼监的态度必须影响第 7 天裁决口径；
- 所有进入结局评分的关键选择必须能追溯到至少一个 StoryEvent。

### 11.4 防止“剧本照读”

固定剧本只提供：

```text
当天必须发生的冲突
可用角色与资源
决策槽位
状态影响边界
下一天入口条件
结局规则
```

最终场景正文、人物对话、信息出现顺序和转场方式由当前 Run Canon 动态生成。

## 12. 主动谋划如何真正影响主线

每次谋划必须产生一个可查询的 `ManeuverOutcome`：

```text
resultFactIds
statePatch
threadOpenedOrTouched
characterMindUpdates
fateSeedId
futureEligibilityChanges
```

示例：使用“清流县令密信”试探商会会首。

```text
立即剧情：商会会首承认见过名册副本，但否认参与篡改。
事实：商会知道名册存在。
关系：商会对总督警惕上升。
线程：名册副本来源进入 open。
未来资格：第 4 天可出现“追查商会保存的副本”。
反噬条件：清算风险过高时，商会公开密信来源。
```

后续 Context Builder 检索到该线程后，AI 才能自然写出“此前那封密信”带来的帮助或反噬。

## 13. API 与任务链路

### 13.1 推荐接口

```text
POST /api/v4/story-runs/:runId/commands/decision
POST /api/v4/story-runs/:runId/commands/maneuver
GET  /api/v4/story-runs/:runId
GET  /api/v4/story-runs/:runId/narrative?afterSequence=120
GET  /api/v4/story-runs/:runId/stream
```

写请求返回：

```json
{
  "accepted": true,
  "taskId": "task_...",
  "runVersion": 18,
  "status": "resolving"
}
```

### 13.2 Worker 阶段

生产形态应由 Worker 执行：

```text
Command accepted
→ Outbox
→ AI Worker
→ NarrativePlan
→ Validate / Repair
→ NarrativeEntry + state transaction
→ Realtime/SSE notification
```

本地 MVP 可以继续同步执行，但必须复用相同的应用服务，避免本地逻辑和 Worker 逻辑两套实现。

### 13.3 幂等与恢复

- 每个 command 必须有 `idempotencyKey`；
- 每个生成任务绑定 `sourceEventId`；
- 同一事件最多存在一个已发布 NarrativeEntry；
- 失败重试复用同一权威状态结果，不重复应用 `statePatch`；
- AI 失败后使用规则 fallback，但仍生成可读的连续剧情；
- 页面刷新后根据 `status=resolving` 恢复流式状态。

## 14. Prompt 设计要求

### 14.1 系统 Prompt 必须强调

```text
你是剧情导演，不是规则制定者。
权威状态结果不可修改。
只写当前玩家能够感知的内容。
必须承接上一场景的时间、地点和未完成动作。
必须体现玩家刚提交的具体行动。
角色只根据已知事实、目标和关系行动。
结尾必须产生给定的下一压力，但不得提前展示选项。
禁止输出内部键、数值补丁、JSON 字段名和技术术语。
```

### 14.2 Writer 文风

- 使用清晰的现代中文历史叙事，不刻意堆砌文言；
- 对话服务于冲突，不做无意义寒暄；
- 每段 40—120 字；
- 先写可见动作，再写合理判断；
- 不直接解释角色完整心理；
- 避免重复“你深知、你意识到、局势发生变化”等模板句；
- 结尾使用具体动作或新消息作为钩子。

## 15. 分阶段实施方案

### Phase 0：立即修复叙事输出边界（1—2 天）

目标：停止明显破坏沉浸感的问题。

- 建立统一 LabelDictionary；
- 禁止内部键进入玩家文本；
- 后端输出单一 `narrative`，前端停止拼接多个回响字段；
- 主线、谋划、关键事件统一走同一结果叙事组件；
- 下一决策只能在结果故事完成并点击继续后显示。

涉及：

```text
apps/api/src/mvp-causal-runtime.ts
apps/web/public/app.js
apps/web/tests/browser-smoke.test.mjs
apps/web/tests/maneuver-ui.test.mjs
```

### Phase 1：连续性上下文与统一 NarrativeEntry（3—5 天）

- 新增 `sequence`；
- 增加 CanonFact、StoryThread、CharacterMind、SceneSnapshot；
- 建立 ContinuityContextBuilder；
- 将现有 messages、visibleCausalCard、roleActions 投影为 NarrativeEntry；
- 保持旧 API 临时兼容。

### Phase 2：Planner / Writer 双阶段（4—7 天）

- 实现结构化 NarrativePlan Schema；
- 规则结果完成后调用 Planner；
- Validator 通过后调用 Writer；
- 增加最多两次 repair；
- DeepSeek 不可用时使用规则连续叙事 fallback。

### Phase 3：跨日线程与角色认知（4—6 天）

- 从十二个主线决策配置初始 StoryThread；
- 决策和谋划更新 CharacterMind；
- FateSeed 与 StoryThread 联动；
- 日终生成结构化 checkpoint；
- 第 2—6 天强制召回相关旧事实。

### Phase 4：异步 Worker 与真实流式输出（5—8 天）

- Command 返回 202；
- Worker 消费 AI 任务；
- SSE 或 Supabase Realtime 发送 NarrativeEntry；
- Worker heartbeat、超时、重试和 Reconciler；
- API 重启后任务可恢复。

### Phase 5：多人视角转译（后续）

- 同一 Canon Event 为不同角色生成不同 NarrativeProjection；
- 每个角色独立 CharacterMind；
- 公共事实、私有事实、误判事实分开；
- 三玩家七轮测试增加跨角色连续性断言。

## 16. 测试与验收标准

### 16.1 单元测试

- Context Builder 对同一状态输出稳定；
- 角色不会读取未知事实；
- StoryThread 到期必定被回收或延期；
- LabelDictionary 覆盖所有公共键；
- `statePatch` 只应用一次；
- NarrativeEntry sequence 严格递增；
- Planner 不能修改权威结果；
- 内部键正则扫描为零命中。

### 16.2 集成测试

对十二个主线决策分别验证：

```text
选择 A/B/C/自定义
→ 状态结果正确
→ 结果故事明确体现选择
→ 至少一个角色反应符合其认知
→ 下一钩子能解释下一 Prompt
→ 刷新后正文和顺序不变
```

### 16.3 七天连续性测试

至少准备四条完整路线：

1. 全面支持改桑；
2. 保护民田并追查暗账；
3. 借商会稳粮；
4. 大量使用自定义决策与主动谋划。

每条路线必须满足：

- 12 次主线决策全部在后续剧情中有可追踪影响；
- 至少 4 次跨日因果回收；
- 第 4 天明确回收第 1—2 天的证据选择；
- 第 6 天奏报引用至少 3 个历史事实；
- 第 7 天裁决能列出事实来源；
- 无时间、地点、人物知识和筹码使用矛盾；
- 无内部键、系统日志和模型术语；
- 下一决策出现前，结果剧情已经展示完毕。

### 16.4 叙事质量自动指标

```text
Action Coverage：正文是否明确体现玩家行动，目标 100%
Fact Grounding：剧情事实有 factId 来源，目标 100%
Knowledge Safety：角色知识越界，目标 0
Internal Key Leakage：内部键泄漏，目标 0
Thread Recall：到期线程未处理，目标 0
Prompt Bridge：下一决策有前置剧情解释，目标 100%
Duplicate Phrase：连续三段模板句重复，目标 0
```

### 16.5 人工试玩问题

每完成一次决策，测试者回答：

```text
我刚才做了什么？
谁因此采取了行动？
当前局势和之前有什么不同？
下一项选择为什么现在出现？
我是否记得至少一条之前留下的线索？
```

五题中任何两题无法回答，本段判定为剧情连续性失败。

## 18. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| Prompt 越来越长 | 成本和延迟失控 | 分层摘要、相关事实检索、日终 checkpoint |
| AI 为了顺畅而篡改事实 | 文案好看但状态矛盾 | 权威结果先生成、Planner 只读、硬校验 |
| 剧情过度模板化 | 每次都是“你深知……” | Writer 风格规则、重复检测、节拍类型轮换 |
| 剧本过于固定 | 用户选择没有意义 | 固定冲突，不固定过程、证据路径和责任解释 |
| 过度自由导致跑题 | 跳日、现代化、角色越权 | World Bible、ActionGuard、必达冲突 |
| 多角色信息泄漏 | 玩家看到隐藏动机 | CharacterMind + 可见事实投影 |
| 状态与正文重复落账 | 数值被加两次 | sourceEventId 幂等、事务、唯一约束 |
| AI 失败导致流程卡死 | 下一决策无法出现 | 规则 fallback + 可恢复任务状态 |

## 19. 架构决策记录（ADR）

### Decision

采用“确定性规则核心 + 连续性上下文 + Planner/Writer 双阶段 AI + 校验与 fallback”的混合叙事架构。

### Drivers

1. 玩家决策必须真实影响后续剧情；
2. 七天主线、角色身份和结局边界不能被 AI 破坏；
3. 剧情必须能跨日记住事实、人物态度和未完成线索。

### Alternatives considered

#### 单次自由生成

优点是开发快、文案自然；缺点是事实漂移、难以回放、难以测试，不适合当前项目。

#### 完全脚本分支树

优点是稳定；缺点是自定义决策和主动谋划无法真正改变剧情，分支成本指数增长。

#### 仅增加长上下文 Prompt

优点是改动小；缺点是没有事实权威、线程管理和角色认知，仍会遗忘或矛盾。

### Why chosen

混合架构可以复用项目现有状态变量、因果账本、FateSeed、DeepSeek 和剧本内容，同时把 AI 限定在最擅长的“情节组织与语言表达”范围内。

### Consequences

- 后端需要新增连续性数据和生成流水线；
- 单次决策可能增加一次 Planner 调用；
- 测试从“接口返回成功”升级为“事实、角色认知、线程和正文一致”；
- 前端反而可以变简单，只消费统一 NarrativeEntry。

### Follow-ups

1. 先完成 Phase 0，关闭内部键和前端拼接问题；
2. 以第 1—2 天四次决策验证 Phase 1—2；
3. 验证通过后扩展到七天；
4. 最后接入 Worker、SSE 和多人视角。

## 20. 最终验收定义

本方案完成后，理想的一次决策体验应是：

```text
玩家读懂当前场景
→ 作出选择或输入自己的行动
→ 系统先确定权威后果
→ AI 根据本局事实、人物认知和未完成线索规划下一段
→ 校验无矛盾后流式展示故事
→ 故事明确承接玩家行动并产生新压力
→ 玩家点击继续
→ 下一决策因为刚刚发生的剧情而自然出现
```

玩家感受到的不是“系统生成了下一张选择题”，而是：

> **我刚才的决定确实改变了这个世界，所以故事才走到了这里。**
