```text
第 1—6 天每天提供 2 次谋划机会，可选使用

```

## 0.0A v4.2 主动谋划系统修订

### v4.3 持续叙事流修订

本版将中间区域从“剧情消息卡片流 + 固定决策面板”升级为“持续叙事时间线”。

正式锁定：

```text
故事连续滚动
→ AI 提供 3 个行动建议
→ 玩家选择建议或输入自己的方案
→ 提交
→ AI 正在推演局势……
→ 新故事自动追加
→ 本次变化自动追加
→ 如有新压力，继续出现下一次决策
```

前台选项只显示标题，不展示收益、风险、成功率和数值预测。

主游戏共 8 个 UI 状态，但只有一个正式路由：

```text
/game?runId=<runId>
```

关键事件回应占用当日 2 次主线决策中的一个槽位，不额外增加单局 12 次主线决策。

本版将主游戏页锁定为两套互补系统：

```text
中间：主线决策系统——回应世界推到玩家面前的关键压力
右侧：主动谋划系统——主动找人、调查、使用筹码、安排幕僚
```

统一术语：

```text
主线决策
主动谋划
谋划机会
自拟谋划
执行谋划
```

第 1—6 天每天提供 2 次谋划机会；第 7 天不提供。谋划可选、未使用不结转，且不能替代每日 2 次主线决策。

禁止继续使用：

```text
AP
行动点
行动力
筹谋
筹谋机会
```

# 1. 产品最终定位

## 1.1 产品不是一个单游戏

本产品不是只做《桑田诏》一个游戏。

《桑田诏：嘉靖财政危局》只是第一个验证用故事局。

底层要设计成平台：

```text
AI 多人局平台
  ├── 故事局大厅
  ├── 多个故事局模板
  ├── 角色系统
  ├── 持续叙事时间线系统
  ├── 玩家决策系统
  ├── AI 推演系统
  ├── 世界状态系统
  ├── 人物关系系统
  ├── 把柄 / 线索系统
  ├── 结局生成系统
  └── 后续多人异步系统
```

后续可以扩展：

```text
历史权谋局
商战局
职场晋升局
悬疑局
家族继承局
董事会逼宫局
创业融资局
宗门权力局
宫廷局
战争外交局
```

---

## 1.2 推荐产品表达

### 对用户

> **AI 多人局：每一场都是一个局。你进入角色，在有限天数内做出关键决策，把局推向不同结局。**

### 对开发团队

> **这是一个 Web 端剧情消息驱动的动态决策游戏。核心不是聊天，不是写小说，而是让玩家通过剧情消息做决策，并实时看到世界状态、人物关系和最终命运发生变化。**

### 对投资人 / 外部沟通

> **AI 多人局是一个 AI 驱动的剧情博弈平台。它把历史、商战、职场、悬疑等复杂局势包装成可玩的“局”，用户通过角色决策改变局势，AI 负责推演后果、生成剧情和结局。**

# 2. 核心产品形态

## 2.1 一句话定义

> **持续叙事时间线驱动的动态决策游戏。**

玩家不是翻阅分页小说，也不是面对消息后台。

玩家看到的是：

```text
连续故事
↓
当前压力
↓
3 个 AI 行动建议 + 统一自定义输入
↓
玩家提交自己的完整决定
↓
AI 正在推演局势
↓
新故事 + 本次变化自动追加
↓
右侧可发起主动谋划
↓
他人影响以角色视角故事进入时间线
↓
最终裁决
```
## 2.2 和旧版“多人 AI 故事”的区别

| 维度 | 旧版多人 AI 故事 | 新版 AI 多人局 |
|---|---|---|
| 用户行为 | 扮演角色提交行动 | 看到剧情压力后做决策 |
| 结构 | 节点 / 回合 / 全员行动后结算 | 天数制 / 持续叙事时间线 / 按消息决策 |
| 中间页面 | 当前剧情 + 行动表单 | 持续叙事时间线 + 当前待决策剧情 |
| 对手行动 | 系统记录或行动列表 | 转译成剧情消息推到你面前 |
| AI 角色 | 可能主动参与 | 只作为局势反应者，不抢主导权 |
| 目标 | 生成一章故事 | 推动一个局走向某种结局 |
| 用户感受 | 我参与了故事 | 我进入了一个局，并改变了局势 |
| 节奏控制 | 节点数 | 天数 + 每日消息数 |
| MVP 优先级 | 小程序 + 多人 | Web + 单人 + AI 机器人 |

本阶段不做小程序。

```text
微信分享能力
小程序登录闭环
小程序支付
小程序订阅消息
```

优先：

```text
Web 游戏主页面
持续叙事时间线
玩家决策
AI 推演
主动谋划与状态反馈
单人 7 天闭环
最终结局
```

---

## 3.2 MVP 只做一个故事局

首个故事局：

```text
《桑田诏：嘉靖财政危局》
```

MVP 控制角色：

```text
玩家控制：浙江总督
AI 角色：浙江巡抚、清流县令、江南商会、司礼监、内阁财政派、皇帝
```

```text
7 天制
```

```text
玩家能完整玩完 7 天
```

# 4. Web MVP 核心体验

## 4.1 玩家真正玩什么

玩家玩的不是剧情选择题，而是：

```text
我是谁？
我现在处在什么局？
别人做了什么？
这件事会不会影响我？
我该压制、交易、密奏、放任，还是借机利用？
我选择之后会改变谁？
我最终会升迁、保命、失势，还是被清算？
```

---

## 4.2 玩家爽点

核心爽点：

```text
1. 我不是旁观者，我是局中人。
2. 别人的行动会变成剧情压力推到我面前。
3. 我可以选系统给的策略，也可以自己写决策。
4. 我的决策会改变世界状态与右侧可执行谋划。
5. 我的决策会变成别人看到的新剧情消息。
6. 我不必只等世界出题，还能主动找人、调查和使用筹码。
7. 主线决策回应眼前危机，主动谋划提前布置后手。
8. 每一天都在逼近最终裁决。
9. 最终下场由主线决策与谋划共同累积而成。
```

# 5. 游戏节奏：天数制，而不是回合制

## 5.1 不叫回合，不叫时段

统一叫：

```text
第 1 天
第 2 天
第 3 天
……
第 7 天
```

示例：

```text
第 3 天 / 共 7 天
地点：杭州总督府
距离御前裁决：4 天
今日主题：粮价与奏疏
```

---

## 5.2 每一天的结构

每一天在同一条中间叙事时间线中完成：

```text
今日开场故事
↓
故事自然推进到第 1 个决策
↓
玩家提交主线决策
↓
AI 正在推演局势
↓
结果故事 + 本次变化
↓
玩家可从右侧发起 0—2 次主动谋划
↓
谋划故事与变化继续追加
↓
第 2 个主线决策
↓
结果故事 + 本次变化
↓
日终故事
↓
进入下一天
```

日终、结果和关键事件都不是独立页面。
## 5.3 每日内容限制

为了避免无限聊天：

```text
每一天最多 3-5 条核心剧情消息
第 1—6 天每天固定 2 次关键玩家决策；第 7 天不再新增普通决策
第 1—6 天每天 2 次谋划机会；第 7 天 0 次；未使用不结转
每条关键决策后最多生成：
  1 条自己的结果消息
  1 条影响他人的剧情消息
  1 条隐藏暗线变化
每天必须进入日终回响
第 7 天必须进入最终裁决
```

# 6. 持续叙事时间线系统

这是主游戏页的核心。

## 6.1 中间主页面是什么

中间区域不是：

```text
系统日志
消息卡片后台
独立结果页
聊天窗口
分页小说
```

中间区域是：

> **一个按时间不断向下生长的角色视角故事。**

它包含：

```text
故事正文
决策模块
AI 推演状态
结果故事
本次变化
关键事件
主动谋划
日终故事
最终裁决
```

所有内容都在中间区域出现。

## 6.2 基本叙事条目

### story

用于开场、角色行动、私密情报、结果故事、日终和结局。

```text
嘉靖三十五年五月初八
杭州 · 总督府

巡抚暮色入府

暮色刚落，巡抚便从西侧小门入内……
```

### decision_prompt

紧接故事出现：

```text
你要如何应对？

A 扣下奏疏
B 秘密召见巡抚
C 放任奏疏入京

[输入你的处理方式……]

提交决策
```

### simulation

```text
你的决定

秘密召见巡抚，并封锁偏厅。

AI 正在推演局势……
```

### change_summary

紧接结果故事自动出现：

```text
本次变化

获得情报：京中势力介入粮局
巡抚敌意 -8
皇帝信任 -3
粮价 +2
```

### critical_event

重要事件先以弹窗提醒；处理后，其故事与回应仍进入中间时间线。

## 6.3 8 个 UI 状态

```text
UI 01 角色专属开场
UI 02 主线故事与决策
UI 03 AI 正在推演
UI 04 推演结果故事与变化
UI 05 局势记录
UI 06 关键事件弹窗
UI 07 他人影响故事与回应
UI 08 主动谋划
```

这些状态不是独立路由，只是中间区域的不同投影。

## 6.4 连续阅读规则

- 不分页；
- 不显示“继续阅读”；
- 不显示“查看变化”；
- 不显示“返回局势”；
- 推演结束自动追加下一段；
- 用户在底部时自动平滑滚动；
- 用户正在向上阅读时不强制跳底；
- 有新内容时显示轻量提示。

## 6.5 视角转译

错误：

```text
浙江巡抚选择了【越级上奏】。
```

正确：

```text
驿站快马离开杭州府。

有人看见巡抚幕僚亲自护送急奏北上。
奏中只写改桑进度，不提粮价与民怨。
```

玩家只能看到当前角色能够感知的事实。

## 6.6 局势记录

“局势记录”默认隐藏，通过中间标题栏的小入口打开。

它只在中间区域展示：

```text
全部
我的决策
他人影响
世界变化
数值变化
```

关闭后恢复原叙事滚动位置。

它不是独立路由，不做复杂因果图，不在其中决策。

## 6.7 关键事件弹窗

仅当事件直接影响玩家且需要回应时弹出：

```text
重要事件发生

巡抚对你的怀疑正在加深

立即处理
稍后处理
```

- 立即处理：进入角色视角故事 + 回应决策；
- 稍后处理：持久化为待处理，不完成决策；
- 普通世界变化不得弹窗；
- 关键事件回应仍占用固定主线决策槽位。
# 7. 玩家决策系统

## 7.1 三个建议 + 一个统一输入框

每个关键决策只展示：

```text
你要如何应对？

A 截留奏疏
B 追加密奏
C 放任巡抚

你也可以写下自己的决定：

[输入你的处理方式……]
0 / 200

提交决策
```

前台不得显示：

```text
可能收益
可能风险
成功率
数值预测
状态补丁
```

这些信息可以存在于后台配置中，但不得进入公共 API。

## 7.2 选择建议后的行为

点击某个建议后：

```text
1. 建议标题写入输入框
2. 玩家可以直接提交
3. 玩家可以继续补充
4. 玩家可以完全改写
```

例如点击“追加密奏”后：

```text
追加密奏，并让县令整理粮价证据作为附件。
```

最终提交的是玩家确认后的完整文本。

## 7.3 自定义决策

玩家可以不选择任何建议，直接输入自己的处理方式。

统一限制：

```text
1—200 字
```

AI 需要判断：

```text
是否符合角色身份
是否符合时代背景
是否拥有当前资源
是否越权
是否直接宣布结果
是否操控其他角色
是否破坏主线
```

## 7.4 ActionGuard

处理结果：

```text
ok
soft_warn
rewrite_needed
blocked
```

被拒绝时：

- 不消耗主线决策；
- 不增加 version；
- 不写入状态补丁；
- 输入内容保留，允许修改。

## 7.5 回应决策

关键事件处理使用同一组件，只改变文案：

```text
提交回应
```

MVP 中回应型 prompt 仍属于当日两次主线决策之一。
# 8. 主动谋划系统

## 8.1 设计目标

主动谋划解决“玩家只能等待系统出题”的问题。

```text
主线决策：回应当前危机
主动谋划：主动调查、交谈、使用筹码、提前布局
```

主动谋划不是第二套主线选择题，也不是无限聊天。

## 8.2 固定规则

```text
第 1—6 天：每天 2 次谋划机会
第 7 天：0 次谋划机会
每次成功谋划使用 1 次
ActionGuard 拒绝不消耗
未使用机会不结转
完成两次主线决策即可进入下一天
主动谋划不能替代主线决策
```

玩家可以在以下状态使用谋划：

```text
awaiting_decision
awaiting_day_advance
```

不能在 `resolving / awaiting_finalization / finished` 状态使用。

## 8.3 谋划类型

### 人物交谈

玩家主动接触当前可接触人物，并选择目的：

```text
试探态度
索取情报
施加压力
提出交易
自拟说辞
```

人物交谈不是自由聊天室。每次交谈必须有明确目标，并产生结构化结果。

### 派遣调查

玩家可派幕僚或可用人员调查：

```text
驿站登记
田契账册
粮仓库存
商会往来
书吏与证人
```

调查可能获得线索，也可能留下被发现的痕迹。

### 使用筹码

玩家将已获得的证据、秘密、人情、承诺、权力或情报用于特定目标。

筹码底层统一使用 `leverage`，前台名称由剧本配置，例如：

```text
历史权谋：筹码
悬疑：线索 / 证据
商战：底牌 / 筹码
生存：物资 / 情报
```

### 自拟谋划

玩家可输入不超过 200 字的主动安排。

前台文案：

```text
自拟谋划
输入你的谋划……
执行谋划
```

必须经过 ActionGuard。

## 8.4 谋划结果

一次成功谋划最多产生：

```text
1 段结果故事
1 组玩家可见变化
1 个 FateSeed 或痕迹
1 条必要的角色反应
```

结果不在右侧形成日志，而是继续追加到中间持续叙事时间线。

前台谋划建议也只显示标题，不显示收益、风险或状态预测。
## 8.5 谋划与主线决策的区别

| 对比项 | 主线决策 | 主动谋划 |
|---|---|---|
| 触发方式 | 世界把问题推给玩家 | 玩家主动出手 |
| 是否必须 | 必须 | 可选 |
| 每日次数 | 固定 2 次 | 最多 2 次 |
| 计数 | `decisionsCompletedToday` | `maneuversUsedToday` |
| 主要影响 | 主线局势与核心因果 | 线索、关系、筹码、任务、伏笔 |
| 是否推进天数 | 两次完成后可推进 | 不能代替主线决策 |
| 结果位置 | 中间叙事时间线 | 同样进入中间叙事时间线 |

## 8.6 谋划闭环

```text
选择人物 / 调查 / 筹码 / 自拟谋划
↓
ActionGuard
↓
扣减 1 次谋划机会
↓
规则引擎 + AI 生成结构化结果
↓
结果写入 StoryEvent
↓
更新 StoryRun 快照
↓
叙事时间线追加结果
↓
影响后续主线、关系、线索、风险和结局
```

# 9. AI 角色定位

## 9.1 AI 角色不是主动玩家

AI 角色不应该像真人玩家一样主动抢胜利。

AI 角色是：

> **局势参与者 / 反应器。**

它们行动只来自三种触发：

```text
1. 玩家影响到它
2. 主线事件推着它
3. 变量阈值触发它
```

---

## 9.2 AI 角色触发示例

```text
粮价 > 70
→ 清流县令请求开仓

巡抚声望被压制
→ 巡抚越级上奏

商会清算风险过高
→ 商会投靠司礼监

皇帝疑心 > 75
→ 司礼监入浙探查
```

---

## 9.3 AI 角色默认惯性

每个 AI 角色有默认倾向。

示例：

```text
浙江巡抚：
默认倾向：抢政绩、快推进、向内阁报功
触发压力：被总督压制、民怨扩大、商会切割
反应方式：越级上奏、拉商会、嫁祸县令

清流县令：
默认倾向：保护百姓、查田契、谨慎上报
触发压力：粮价过高、百姓请愿、总督不保护
反应方式：递密信、请开仓、保留账册副本

江南商会：
默认倾向：保现金流、拿政策特权、避免清算
触发压力：被查账、粮价失控、官府逼粮
反应方式：放粮交易、投靠司礼监、转移账册
```

## 15.3 Prisma 建议

```prisma
model StoryRun {
  id              String       @id @default(cuid())
  templateKey     String
  userId          String?
  mode            String       @default("single")
  status          String       @default("created")
  currentDay      Int          @default(1)
  totalDays       Int          @default(7)
  selectedRoleKey String       @default("zhejiang_governor")
  stateJson       Json
  version         Int          @default(1)
  createdAt       DateTime     @default(now())
  updatedAt       DateTime     @updatedAt
  events          StoryEvent[]

  @@index([templateKey])
  @@index([userId])
  @@index([status])
}

model StoryEvent {
  id          String   @id @default(cuid())
  runId       String
  run         StoryRun @relation(fields: [runId], references: [id], onDelete: Cascade)
  day         Int
  type        String
  messageType String?
  roleKey     String?
  visibility  String   @default("private")
  payloadJson Json
  createdAt   DateTime @default(now())

  @@index([runId, day, createdAt])
  @@index([runId, type])
  @@index([runId, messageType])
}
```

`stateJson` 保存 worldState、roleState、relationships、risks、clues、fateSeeds、evidenceLedger、responsibilityLedger、narrativeFrames、roleDecisionModels、daySummaries 和 finalJudgementInputs。

# 22. 最终判断

本产品当前阶段最重要的不是“AI 能不能写出漂亮小说”。

最重要的是验证：

> **用户是否愿意在持续叙事时间线中不断做决策，并因为看到局势变化而继续玩下去。**

一句话：

> **AI 多人局不是单个游戏，而是一个可扩展的 AI 动态局势推演平台；Web MVP 先用《桑田诏》验证“剧情消息 → 主线决策 → 主动谋划 → 状态变化 → 新消息 → 最终裁决”的核心循环。**
