# AI 多人剧情推演：主游戏页持续叙事流 UI / 交互设计文档（MVP）

> 文档用途：  
> 1. 作为 Codex 开发主游戏页面的产品与交互规格。  
> 2. 作为 ChatGPT / 设计工具生成 8 张独立 UI 设计图的统一依据。  
> 3. 作为后续功能验收的页面标准。  
>
> 视觉基准：以用户提供的“杭州总督府 · 内厅”主游戏页面为唯一参考。顶部、左栏、右栏、整体比例、字体、边框、米白宣纸、水墨纹样、朱红与金色按钮等视觉语言必须保持一致。

---

## 1. 产品页面的核心定义

主游戏不是“消息后台”，也不是“多个功能页面之间来回跳转”。

它应该是一条持续生长的互动叙事时间线：

```text
故事发生
  ↓
AI 给出几个行动建议
  ↓
玩家选择建议或输入自己的决定
  ↓
提交决策
  ↓
AI 正在推演局势
  ↓
下一段故事自然出现
  ↓
数值与状态变化自动展示
  ↓
新的故事、新的影响、新的决策继续出现
```

玩家始终停留在同一个主游戏页面。

页面的核心分工：

```text
左侧：我是谁、我拥有什么、我面临什么风险
中间：世界正在发生什么、我如何回应、结果如何继续
右侧：我还可以主动接触谁、使用什么筹码、发起什么谋划
```

最重要的页面原则：

> 所有故事、事件、AI 建议、玩家决策、推演状态、结果变化、他人影响，都只在中间区域展示。  
> 顶部、左侧身份栏、右侧谋划栏始终保持原有页面结构与视觉风格。

---

## 2. 页面实现方式

### 2.1 设计稿数量

MVP 需要设计 **8 张独立 UI 图**。

这 8 张图代表 8 种主要页面状态，但开发时不是 8 个独立路由。

### 2.2 开发页面数量

开发层面建议只有一个主路由，例如：

```text
/game/[roomId]
```

顶部、左栏、右栏保持固定，中间区域根据状态切换或追加内容。

### 2.3 中间区域的基本内容类型

中间叙事流只需要以下几种内容模块：

```text
StoryBlock          故事正文
DecisionComposer    决策与自定义输入
SimulationBlock     AI 推演中
ChangeSummary       数值与状态变化
SituationRecord     局势记录
CriticalEventModal  关键事件弹窗
ActivePlanBlock     主动谋划
```

---

## 3. 固定页面骨架

8 张 UI 图必须共用同一个完整页面骨架。

### 3.1 顶部区域：始终不变

继续保留参考页面中的信息：

- 当前地点：杭州总督府 · 内厅
- 当前日期与时段
- 距离御前裁决剩余天数
- 今日主线决策次数
- 今日谋划次数
- 国库银
- 民心
- 粮价
- 改革进度
- 皇帝信任
- 历史回顾入口
- 设置入口

MVP 中“历史回顾”不设计独立页面，可暂时保留视觉入口；需要回看内容时，统一使用中间区域的“局势记录”。

### 3.2 左侧区域：始终不变

左栏回答“我是谁”。

固定展示：

- 我的身份
- 角色头像
- 姓名与官职
- 品级
- 命运问题
- 当前目标
- 我的资源
- 我的筹码
- 当前风险

故事、推演、弹窗、主动谋划过程中，左栏不切换、不隐藏、不变成其他功能。

### 3.3 右侧区域：始终不变

右栏回答“我还能做什么”。

固定展示：

- 今日可用谋划次数
- 可接触人物
- 人物关系或当前态度
- 可执行动作
- 正在推进的目标
- 可用筹码
- 自拟谋划输入
- 进入下一天

点击右侧人物动作或提交自拟谋划后，具体内容必须进入中间区域展示。

### 3.4 中间区域：唯一变化区域

中间区域负责：

- 开场故事
- 当前剧情
- AI 给出的建议
- 玩家自定义决策
- AI 推演状态
- 推演后的新剧情
- 本次数值与状态变化
- 其他玩家造成的影响
- 主动谋划
- 局势记录

## 4. 核心交互规则

### 4.1 持续叙事流，不做分页阅读

取消：

- 1 / 4
- 继续阅读
- 下一页
- 上一页
- 查看本次变化
- 返回局势纪事
- 继续行动
- 完成
- 决定如何回应
- 进入决策

故事采用中间区域内部连续滚动：

```text
故事第一段
故事第二段
故事第三段
决策模块
AI 推演中
下一段故事
本次变化
下一次决策
```

### 4.2 决策选项只显示行动名称

AI 建议中不显示：

- 可能收益
- 可能风险
- 后果预测
- 成功概率
- 数值变化预测
- 复杂说明

正确形式：

```text
你要如何应对？

A 扣下奏疏
B 秘密召见巡抚
C 放任奏疏入京

你也可以写下自己的决定：
[输入框]

提交决策
```

### 4.3 AI 建议与自定义决策统一

用户点击 AI 建议后，将建议内容写入输入框。

例如点击：

```text
B 秘密召见巡抚
```

输入框自动出现：

```text
秘密召见巡抚
```

用户可以直接提交，也可以补充：

```text
秘密召见巡抚，并要求两名亲信封锁偏厅，
不得允许任何随从靠近。
```

用户也可以完全不选择 AI 建议，直接输入自己的方案。

最终提交的永远是一段完整的玩家决定。

### 4.4 AI 推演状态极简

只展示：

```text
你的决定

秘密召见巡抚

AI 正在推演局势……
```

可以加入：

- 印章轻微呼吸
- 水墨缓慢晕染
- 省略号动画
- 轻微纸面光影

不展示：

- 正在判断谁是否配合
- 正在计算关系
- 正在分析其他玩家
- 权重
- 概率
- 随机数
- 系统推理过程
- 后端任务步骤

### 4.5 推演完成后自动追加内容

AI 推演完成后，不需要用户点击任何按钮。

中间区域自动出现：

```text
下一段故事
↓
本次变化
↓
新的决策（如果当前剧情需要）
```

### 4.6 数值变化只展示变化本身

变化区只展示玩家当前允许知道的内容，例如：

```text
本次变化

获得情报：京中势力介入杭州粮局
巡抚信任：+8
皇帝信任：-3
行动力：-1
```

不需要：

- 单独结果页面
- 返回按钮
- 继续行动按钮
- 查看完整因果链
- 查看其他玩家具体数值
- 复杂系统说明

### 4.7 其他玩家影响必须遵守视角限制

不能直接告诉当前玩家：

```text
玩家 A 选择了向巡抚举报你。
```

应该转化为当前角色能感知的故事：

```text
一封没有署名的密报被送入巡抚府。

密报准确写出了商会账本被扣留的时间，
甚至提到了只有你和两名幕僚知道的暗格位置。
```

当前玩家只能知道：

- 自己看到的事实
- 自己收到的消息
- 自己能够合理推断的信息

不能知道：

- 对方点击了哪个选项
- 对方真正的目标
- 对方的隐藏任务
- 对方未公开的数值变化
- 对方的完整行动过程

## 5. 8 个 UI 页面总览

| 编号 | UI 名称 | 中间区域状态 | 是否为独立路由 |
|---|---|---|---|
| UI 01 | 角色专属开场 | 纯故事连续阅读 | 否 |
| UI 02 | 主线故事与决策 | 故事结尾 + AI 建议 + 自定义决策 | 否 |
| UI 03 | AI 正在推演 | 已提交决定 + 极简推演状态 | 否 |
| UI 04 | 推演结果故事与变化 | 新剧情 + 自动数值变化 + 可选下一决策 | 否 |
| UI 05 | 局势记录展开 | 中间区域回看历史事件 | 否 |
| UI 06 | 关键事件弹窗 | 主页面压暗 + 中央弹窗 | 否 |
| UI 07 | 他人影响故事与回应 | 他人影响故事 + AI 回应建议 + 自定义回应 | 否 |
| UI 08 | 主动谋划 | 谋划目标 + AI 建议 + 自拟谋划 | 否 |

## 6. UI 01：角色专属开场故事

### 6.1 使用场景

- 玩家第一次进入游戏
- 新的一天开始
- 新章节开始
- 重大剧情阶段开始

### 6.2 页面目标

让玩家立即进入自己的角色视角，并明确：

- 我是谁
- 我现在在哪里
- 当前世界发生了什么
- 我面临的核心矛盾是什么

### 6.3 中间区域结构

中间顶部：

```text
嘉靖三十五年五月初八
杭州 · 总督府

第一章：杭州粮局
```

正文区域：

```text
杭州粮价已经连续上涨三日。

城中米行陆续闭门，百姓开始聚集在粮铺之外。
巡抚的奏疏在午前送到总督府，奏疏中将粮价失控
归因于江南商会囤积居奇。

但你知道，事情远没有这么简单。

昨夜送来的密报里，出现了司礼监的名字。
```

### 6.4 页面结构

```text
[中间标题栏]
当前章节标题                              局势记录

[日期 / 地点]
[章节标题]

[连续故事正文]
[连续故事正文]
[连续故事正文]

[继续向下自然出现后续故事或 UI 02 决策模块]
```

### 6.5 交互要求

- 不显示“继续阅读”。
- 不显示页码。
- 不显示决策选项，除非故事已经自然推进到决策节点。
- 内容不足一屏时，保持大面积留白与水墨背景。
- 内容超出一屏时，中间区域独立滚动。
- 左右栏与顶部固定不动。

### 6.6 设计图必须表现

- 完整主页面骨架
- 左右栏不变
- 中间为故事阅读状态
- 没有消息卡片
- 没有管理后台感
- 没有额外流程按钮

---

## 7. UI 02：主线故事与决策

### 7.1 使用场景

故事推进到当前角色必须作出决定的位置。

### 7.2 页面目标

让玩家感受到：

> 决策是故事的一部分，而不是从故事突然跳到一个功能面板。

### 7.3 中间区域上半部分：故事结尾

```text
巡抚的奏疏已经摆在案前。

奏疏中并未否认杭州粮价失控，
却将责任全部指向江南商会。

若这份奏疏送入京师，皇帝将先看到巡抚的一面之词。

现在，你必须作出决定。
```

### 7.4 中间区域下半部分：决策模块

标题：

```text
你要如何应对？
```

AI 建议：

```text
A 扣下奏疏
B 秘密召见巡抚
C 放任奏疏入京
```

自定义输入：

```text
你也可以写下自己的决定：

[输入你的处理方式……]
0 / 200
```

提交按钮：

```text
提交决策
```

### 7.5 选项样式

选项只包含：

- A / B / C 编号
- 行动名称

不包含：

- 说明段落
- 收益
- 风险
- 数值
- 预测后果

建议采用与参考图一致的古典卡片边框，但卡片高度应明显降低，避免像复杂功能卡。

### 7.6 选择后的行为

用户点击 AI 建议：

```text
B 秘密召见巡抚
```

输入框自动填入：

```text
秘密召见巡抚
```

用户可补充或修改，然后点击“提交决策”。

### 7.7 提交按钮状态

未输入内容时：

```text
提交决策（禁用）
```

输入或选择方案后：

```text
提交决策（可用）
```

提交期间：

```text
提交中……
```

必须防止重复提交。

## 8. UI 03：AI 正在推演

### 8.1 使用场景

玩家提交：

- 主线决策
- 他人事件回应
- 主动谋划

### 8.2 页面目标

用非常短的过渡状态告诉玩家：

> 决定已经被系统接收，世界正在根据所有角色与状态继续发展。

### 8.3 中间区域结构

```text
你的决定

秘密召见巡抚

你命人从侧门召巡抚入府，
并要求偏厅周围不得留下无关人员。

AI 正在推演局势……
```

### 8.4 视觉表现

建议：

- 中央印章图形
- 印章轻微呼吸
- 水墨从中心缓慢扩散
- “AI 正在推演局势……”省略号动画
- 当前决定以古代公文或批注形式展示

### 8.5 不允许出现

- 正在判断巡抚是否愿意配合
- 正在评估官场势力
- 正在分析是否影响其他玩家
- 加载进度百分比
- 模型名称
- Token 信息
- 系统计算过程
- 权重与概率

### 8.6 状态结束

推演完成后，中间区域自动进入 UI 04。

不出现：

- 查看结果
- 下一步
- 继续
- 完成

## 9. UI 04：推演结果故事与变化

### 9.1 使用场景

AI 完成推演后，展示玩家决定造成的新剧情与可见变化。

### 9.2 页面目标

顺序必须是：

```text
先让玩家看到发生了什么
再让玩家看到数值发生了什么变化
最后在需要时自然出现下一次决策
```

### 9.3 中间区域第一部分：结果故事

```text
巡抚暮色入府

暮色刚刚落下，巡抚便从总督府西侧的小门入内。

他没有携带随从，只在袖中藏着一封尚未加印的奏疏。

“总督大人，杭州粮价失控，并非商会一方所为。”

他停顿片刻，压低了声音。

“京中有人，希望杭州先乱起来。”

就在此时，偏厅外传来一阵急促的脚步声。
```

### 9.4 中间区域第二部分：本次变化

故事正文下方自动出现：

```text
──────── 本次变化 ────────

获得情报
京中势力正在介入杭州粮局

巡抚信任      +8
皇帝信任      -3
行动力        -1
```

变化模块只展示玩家当前可以知道的变化。

### 9.5 可见性规则

可以展示：

- 自己的资源变化
- 自己的关系变化
- 自己获得的情报
- 自己能够观察到的世界变化
- 已经公开发生的其他角色影响

不能展示：

- 其他玩家隐藏数值
- 其他玩家隐藏目标
- 未公开的具体后果
- 系统内部因果链
- “玩家张三声望 -10”

### 9.6 后续剧情

如果当前故事立即出现新的问题，则在变化模块下方继续追加：

```text
门外的人即将进入偏厅。

你要如何应对？

A 让亲信将来人拦在门外
B 让巡抚暂时藏入侧室
C 直接开门查看来人身份

[输入你的决定……]

提交决策
```

如果暂时没有新决策，中间区域停留在最新内容，不显示任何完成按钮。

## 10. UI 05：局势记录展开状态

### 10.1 使用场景

玩家需要回看：

- 之前发生过什么
- 自己做过什么决定
- 哪些事件影响了自己
- 哪些数值发生了变化

### 10.2 打开方式

中间标题栏右上角保留小型辅助入口：

```text
局势记录
```

它不应成为醒目的主按钮。

示例：

```text
杭州粮局                              局势记录
```

### 10.3 页面行为

点击“局势记录”后：

- 顶部不变
- 左栏不变
- 右栏不变
- 只切换中间区域
- 保存原叙事流滚动位置
- 关闭记录后恢复原来的阅读位置

### 10.4 中间区域结构

标题：

```text
局势记录
```

MVP 筛选：

```text
全部
我的决策
他人影响
世界变化
数值变化
```

事件列表：

```text
午后

你秘密召见浙江巡抚

巡抚透露，京中某股势力可能介入杭州粮局。

巡抚信任 +8
皇帝信任 -3
```

```text
未时

城内出现抢粮风潮

米行门前开始聚集百姓，部分粮铺提前闭门。

民心 -4
粮价 +3
```

```text
申时

两日前释放粮商的决定产生后果

几名粮商重新开仓，但向官府提出了新的条件。
```

### 10.5 局势记录只负责回看

不在局势记录中：

- 做决策
- 显示复杂因果图
- 展示人物关系网
- 展示其他玩家隐藏数据
- 打开独立详情页
- 跳转新路由

### 10.6 关闭方式

可以使用中间标题栏中的：

```text
返回当前故事
```

这是局势记录辅助视图的关闭入口，不属于叙事流程按钮。

若希望进一步减少按钮，也可将“局势记录”做成可再次点击关闭的切换项。

## 11. UI 06：关键事件弹窗

### 11.1 使用场景

仅用于真正直接影响当前玩家的重要事件：

- 另一名玩家公开弹劾当前玩家
- 当前玩家的密信被截获
- 朝廷开始调查当前玩家
- 盟友倒戈
- 商会因他人的决策停止供粮
- 当前计划因他人行动被打断
- 当前角色面临必须回应的危机

普通世界变化不能弹窗。

### 11.2 页面形式

- 完整主页面保留
- 页面整体变暗
- 中央出现古典纸张式弹窗
- 左右栏和顶部仍可隐约看见

### 11.3 弹窗内容

```text
重要事件发生

巡抚对你的怀疑正在加深

一封密报已经被送入巡抚府。
其中提到了你曾私下扣留商会账本。

立即处理
稍后处理
```

### 11.4 两个按钮

关键事件弹窗是唯一允许使用流程双按钮的场景：

```text
立即处理
稍后处理
```

点击“立即处理”：

- 关闭弹窗
- 中间区域进入 UI 07

点击“稍后处理”：

- 关闭弹窗
- 将该事件加入局势记录
- 在当前叙事流顶部或最新位置显示“待处理”提示
- 不丢失事件
- 需要时再次弹出或允许用户从待处理提示进入

### 11.5 弹窗限制

避免频繁打断。

建议只有满足以下条件之一才弹出：

```text
requires_response = true
is_directly_affected = true
severity >= high
```

## 12. UI 07：他人影响故事与回应

### 12.1 使用场景

### 12.2 页面目标

把其他玩家或世界事件的影响转化为当前角色视角下的一段故事，然后自然进入回应决策。

### 12.3 中间区域第一部分：角色视角故事

```text
一封没有署名的密报
```

- 对方为什么这么做
- 对方的目标

## 17. 滚动与新内容规则

### 17.1 默认自动滚动

- 自动平滑滚动到新故事
- 不产生页面整体跳动
- 顶部、左栏、右栏保持固定

- 不强制跳到底部

- 用户点击后回到最新内容

该提示属于滚动辅助，不是剧情流程按钮。

打开局势记录时保存：

```ts
narrativeScrollTop
```

关闭后恢复原位置。

### 18.1 提交锁定

- 禁用 AI 建议
- 禁用提交按钮

```text
局势推演暂时中断

你的决定已经保存，但新的剧情尚未生成。

```

允许：

```text
重新推演
```

错误恢复按钮属于异常处理，不计入正常流程按钮。

### 18.3 网络恢复

重新推演时必须复用同一个 `decisionId`，防止重复扣除：

- 主线决策次数
- 谋划次数
- 行动力
- 筹码

## 19. 视觉设计规范

### 19.1 总体风格

必须延续参考页面：

- 米白宣纸背景
- 淡水墨山水
- 古典宋体 / 楷体标题
- 深墨正文
- 朱红重要操作
- 金色次级操作
- 青绿正向状态
- 朱红负向状态
- 细线古典边框
- 四角回纹装饰
- 轻微纸张纹理
- 低饱和度人物头像

### 19.2 不允许出现

- 现代 SaaS 卡片风格
- 纯白大面积后台面板
- 蓝紫渐变
- 霓虹色
- 现代聊天气泡
- 移动端样式
- 黑色全屏故事页
- 过重阴影
- 圆角过大
- 玻璃拟态
- 游戏商城风格
- 不同页面改变整体比例

### 19.3 中间区域阅读排版

建议：

- 正文不铺满整个中间宽度
- 保留左右呼吸空间
- 段间距明显
- 行距舒适
- 标题居中或略偏古籍排版
- 日期地点弱化
- 决策模块与故事之间使用细分隔纹样
- 变化模块紧接故事，但视觉权重低于正文

### 19.4 8 张 UI 图的一致性

每张设计图必须使用完全相同的：

- 画布比例
- 顶部高度
- 左栏宽度
- 右栏宽度
- 中间宽度
- 字体体系
- 人物头像
- 世界数值
- 背景纹样
- 边框样式
- 按钮样式

只能改变中间区域状态；UI 06 额外增加中央弹窗。

## 24. 最终产品体验定义

玩家进入页面以后，体验应该始终是：

```text
我正在阅读属于自己的故事
  ↓
故事把我推到一个必须选择的位置
  ↓
AI 给我几个可以参考的行动方向
  ↓
我作出并提交自己的决定
  ↓
AI 根据所有角色、世界状态与之前行动继续推演
  ↓
我看到新的故事和真实变化
  ↓
其他玩家的行动也可能进入我的故事
  ↓
我继续回应、谋划和改变这个世界
```

最终定义：

> 中间区域不是消息列表，也不是功能页面集合，而是一条持续生长的叙事时间线。故事负责告诉玩家发生了什么，AI 负责提供可参考的行动方向，玩家负责作出最终决定，系统再把所有人的选择推演成新的世界。
