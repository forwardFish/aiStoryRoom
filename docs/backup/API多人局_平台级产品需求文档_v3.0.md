# AI 多人局｜平台级产品需求文档 v3.0

> 文档类型：产品需求文档 / 平台级 PRD  
> 版本：v3.0  
> 核心方向：从「多人 AI 故事」升级为「AI 多人局」平台  
> MVP 示例故事局：《桑田诏：嘉靖财政危局》  
> 目标读者：产品、前端、后端、AI 工程、设计、投资人沟通  
> 输出目的：统一定位、玩法、系统、页面、AI 推演、数据结构、风险与迭代路线

# 0. 本次更新说明

## 0.1 为什么要更新

历史文件中的核心方向是「多人 AI 故事 / 多人命运线共演」，重点解决的问题是：

- 多个玩家如何进入同一个故事世界；
- 每个人如何有自己的目标、秘密和命运线；
- 玩家如何感知自己的行动影响别人和世界；
- AI 如何把多条故事线交织成最终小说；
- 页面如何让玩家看到“我不是配角，我是这个世界的关键人物”。

这个方向是对的，但如果继续停留在「多人 AI 故事」，容易出现几个问题：

1. **产品像 AI 写故事工具，而不像游戏。**  
2. **用户容易变成读剧情，而不是做决策。**  
3. **多人交互容易变成聊天房，缺少明确目标和局势压力。**  
4. **系统容易无限发散，成本和体验都失控。**  
5. **如果只围绕一个剧本设计，后续扩展多个游戏会困难。**

因此，本版本将产品升级为：

> **AI 多人局平台。**

即：

> 每一场都是一个“局”。玩家进入一个有限天数、有限角色、有限目标、有限资源、有限风险的 AI 推演局中，和真人玩家或 AI 角色共同博弈，通过剧情消息做决策，最终看自己把局推向什么结局。

## 0.2 本次核心变化

| 模块 | 历史版本 | 最新版本 v3.0 | 更新原因 |
|---|---|---|---|
| 产品定位 | 多人 AI 故事 / 多人命运线共演 | AI 多人局平台 | 用户和投资人更容易理解“每一场是一个局”，不是单纯写故事 |
| 核心体验 | 玩家进入同一故事世界，影响别人故事线 | 玩家进入一个局，通过剧情消息做决策，改变局势和命运 | 强化游戏感、博弈感、决策感 |
| 推进方式 | 故事线交织 / 回合回响 | 固定天数 + 剧情消息流 + 玩家决策 + 局势反馈 | 避免无限聊天，增强节奏和压力 |
| 页面核心 | 命运网、故事线、公共/私密信息 | 中间剧情消息流 + 当前决策区，左右只做状态辅助 | 中间必须是游戏现场，而不是信息后台 |
| 角色状态追踪 | 基础支持 | 深度支持，含命运轨道 / 风险 / 把柄 / 关系变量 | 让玩家感知“我正在改变自己的下场” |
| AI 推演方式 | 主线推进 | 动态局势推演 + 玩家插手 + AI 角色反应 | AI 不只是写剧情，而是推演局势 |
| AI 角色定位 | 可能像玩家一样参与 | AI 角色是局势参与者，不主动抢戏 | 单人可玩，多人不被 AI 抢主导权 |
| 多人机制 | 多人共同参与故事 | 多人异步行动，行动被转译为对他人的剧情消息 | 不要求所有人同时在线，避免卡局 |
| 单人模式 | 非重点 | MVP 先做单人，其他角色由 AI 机器人反应 | 降低冷启动难度 |
| 扩展性 | 单项目倾向 | 平台化，支持多个故事局 | 后续可做历史、商业、职场、悬疑等内容库 |
| 商业化 | 单剧本体验 | 平台 + 内容库 + 付费局 + 高级剧本 + 多人房间 | 商业模式更清晰 |

# 1. 产品最终定位

## 1.1 推荐名称

本产品需要一个用户和投资人都能听懂的名字。

### 候选名称对比

| 名称 | 优点 | 问题 | 适合用途 |
|---|---|---|---|
| 多人 AI 故事 | 容易理解有 AI、有故事、多人 | 太像写故事工具，游戏感弱 | 早期解释，不建议作为最终定位 |
| 多人 AI 故事局 | 比“故事”更有局势感 | 名字偏长，传播略弱 | PRD 内部描述可用 |
| 多人 AI 局 | 强调“多人”和“局” | 可能被理解成多个 AI，而不是多人玩家 | 可作为玩法描述 |
| AI 多人局 | 简短、有传播性，AI 在前，用户容易理解 | 需要用副标题解释“局”是什么 | 推荐作为品类名 |
| AI 命运局 | 有故事感和结局感 | 不够明确多人属性 | 可作为某个产品名或剧本分类 |
| AI 剧情博弈 | 投资人容易理解博弈属性 | 用户侧略硬 | 对外路演可用 |
| AI 互动局 | 泛化强 | 不够有差异化 | 可作为二级描述 |

## 1.2 推荐最终表达

### 对用户说

> **AI 多人局：每一场都是一个局。你进入角色，在固定天数内做出决策，和 AI 或真人一起把局推向不同结局。**

### 对投资人说

> **AI 多人局是一个剧情消息驱动的动态博弈游戏平台。它用 AI 生成和推演局势，让用户在历史、商战、职场、悬疑等不同故事局中进行多人或单人决策博弈，形成高复玩、多内容扩展的互动娱乐产品。**

### 对开发团队说

> **这是一个平台型网页游戏。底层是可配置故事局引擎；前端核心是剧情消息流 + 决策面板 + 局势反馈；后端核心是状态机 + AI 推演编排 + 多人消息同步。**

# 2. 产品类型判断

## 2.1 它属于什么类型的游戏？

它不是传统文字游戏，也不是剧本杀，也不是纯策略游戏。

更准确的品类是：

> **AI 驱动的剧情博弈游戏。**

也可以描述为：

```text
AI Narrative Strategy Game
AI Social Simulation Game
AI Dynamic Story Game
剧情消息驱动的多人博弈游戏
动态局势推演网页游戏
```

## 2.2 为什么不能让用户觉得是“文字游戏”

如果用户觉得这是文字游戏，会降低期待：

- 以为只是读文本；
- 以为只是选 A/B/C；
- 以为没有真实互动；
- 以为 AI 只是写故事。

所以产品必须突出：

```text
剧情消息 + 我的决策 + 局势反馈
```

核心不是“阅读”，而是：

> **我看到剧情压力，然后做决策，决策改变局势，局势反过来压迫我。**

这才是互动游戏。

# 3. 产品一句话与核心卖点

## 3.1 产品一句话

> **AI 多人局：进入一个局，在有限天数内做关键决策，和 AI 或真人共同把局推向不同结局。**

## 3.2 核心卖点

1. **不是看故事，而是入局。**  
   玩家不是旁观者，而是角色本人。

2. **每条剧情消息都可能改变局势。**  
   巡抚上奏、商会变卦、县令递密信、司礼监入场，都会推到玩家面前。

3. **玩家可以选择，也可以自定义决策。**  
   不只有固定选项，还可以写自己的策略，但 AI 会校验是否合理。

4. **AI 不只是写剧情，而是推演后果。**  
   玩家选择会改变世界变量、人物关系、把柄流向和最终结局。

5. **单人可玩，多人更强。**  
   单人时 AI 角色反应；多人时真人行动互相转译成剧情压力。

6. **平台化扩展。**  
   一个引擎可以支持历史、商战、职场、悬疑等多个故事局。

# 4. 历史需求文件核心整理

历史文件的核心思想可归纳为以下 8 点。

## 4.1 可感知的交织机制

历史文件强调：难点不是让 AI 写剧情，而是让用户清楚感知到：

> 我不是在公共故事里当配角，而是我的选择正在改变别人、改变世界、改变最终小说。

这点在最新版本中保留，并升级为：

> 玩家每次决策都必须产生可见的局势反馈，并可能生成推送给其他角色的新剧情消息。

## 4.2 命运钩子

历史版本提出：每个玩家从不同入口进入同一个核心冲突。

最新版本继承这个思想，并将其抽象为故事局模板字段：

```text
core_conflict：核心冲突
role_entry_points：角色入口
role_private_motivation：角色私密动机
role_fate_question：角色命运问题
```

## 4.3 谋略感的 5 个来源

历史版本指出谋略感来自：

```text
信息不对称
目标不完全一致
角色之间互相影响
行动有延迟后果
公开信息和私密信息不断变化
```

最新版本把这 5 点产品化为：

```text
私密消息
角色目标
关系变量
隐藏暗线
把柄系统
剧情消息可见性
延迟后果状态
```

## 4.4 玩家行动必须影响别人

历史版本强调：玩家 A 的选择不能只影响玩家 A。

最新版本升级为：

```text
玩家决策后，最多生成：
1 条自己的结果反馈
1 条影响他人的剧情消息
1 条隐藏暗线变化
```

这样可以保证每次关键决策都有交织感，但又不无限发散。

## 4.5 行动要有延迟后果

历史版本提到：真正有趣的是现在做的事，可能两轮后才暴露后果。

最新版本改成：

```text
hidden_threads
risk_flags
delayed_messages
future_triggers
```

例如：

```text
今日你密奏皇帝。
第 5 天司礼监发现奏报差异。
第 6 天巡抚反咬你越级自保。
```

## 4.6 公开信息和私密信息变化

历史版本强调页面上要区分：

```text
我知道的
大家知道的
别人可能知道但没有说的
```

最新版本落到消息系统中：

```text
visibility: public / private / faction / hidden
```

## 4.7 命运网

历史版本提出“命运网”，让玩家看到故事线如何互相牵连。

最新版本将命运网简化为开发可落地的几个模块：

```text
人物关系
把柄流向
影响消息
隐藏暗线
最终命运结算
```

不建议 MVP 一开始做复杂可视化命运网，先用右侧人物关系和最新变化承载。

## 4.8 最终小说与个人故事卡

历史版本提出最终应生成多 POV 小说和个人故事卡。

最新版本保留，但作为第二阶段增强功能：

```text
MVP：先做个人结局 + 全局结局
V2：增加本局小说
V3：增加可分享个人命运卡
```

# 6. 最新产品架构

## 6.1 平台结构

```text
AI 多人局平台
  ├── 故事局大厅
  ├── 故事局详情页
  ├── 角色选择页
  ├── 主游戏页
  │     ├── 左侧：我的信息
  │     ├── 中间：剧情消息流 + 当前决策
  │     └── 右侧：局势反馈
  ├── 日终回响页
  ├── 最终结局页
  └── 历史记录 / 分享页
```

## 6.2 故事局模板结构

每个故事局都应由配置驱动：

```text
story_game_template:
  title
  category
  visual_theme
  total_days
  player_count_range
  world_background
  core_conflict
  day_plan
  roles
  world_variables
  relationship_variables
  clue_types
  message_rules
  decision_rules
  ending_rules
```

## 6.3 可复用平台能力

以下能力必须与具体剧本解耦：

```text
房间创建
玩家入局
角色绑定
消息流展示
决策提交
AI 推演
状态更新
历史记录
结局生成
多剧本配置
```

# 7. MVP 设计

## 7.1 MVP 目标

MVP 目标不是一次性做完整多人平台。

MVP 目标是验证：

> 玩家是否愿意在“剧情消息 + 我的决策 + 局势反馈”的循环里持续玩完一个局。

## 7.2 MVP 范围

第一版只做一个故事局：

```text
《桑田诏：嘉靖财政危局》
```

第一版只做单人模式：

```text
玩家控制：浙江总督
AI 角色：浙江巡抚、清流县令、江南商会、司礼监、内阁、皇帝
```

第一版核心功能：

```text
1. 故事局大厅
2. 剧本详情页
3. 角色确认页
4. 主游戏页
5. 7 天制剧情消息流
6. 每条关键消息 3 个选项 + 自定义输入
7. AI 推演状态变化
8. 右侧局势反馈
9. 日终回响
10. 最终个人结局 + 全局结局
```

## 7.3 MVP 不做什么

MVP 暂时不做：

```text
复杂多人实时同步
复杂命运网可视化
长篇最终小说生成
开放地图探索
语音
战斗系统
复杂动画
UGC 剧本编辑器
```

# 8. 核心玩法循环

最新版本的核心玩法循环为：

```text
当天开局剧情
↓
剧情消息流出现压力
↓
玩家看到关键消息
↓
系统给出 3 个选项 + 自定义决策
↓
玩家决策
↓
AI 校验是否合法
↓
AI 推演后果
↓
右侧更新世界状态 / 人物关系 / 风险 / 把柄
↓
生成新剧情消息或隐藏暗线
↓
日终回响
↓
进入下一天
↓
最终裁决
```

这个循环必须稳定、短、清晰。

不要变成无限聊天。

# 9. 主游戏页设计要求

## 9.1 页面总结构

主页面采用三栏结构：

```text
左侧：我手里的牌
中间：游戏现场
右侧：世界反馈
```

## 9.2 左侧：我的信息

左侧只放帮助玩家做决策的信息：

```text
我的身份
我的目标
我的资源
我的筹码
我的当前风险
```

示例：

```text
【我的身份】
浙江总督

【我的目标】
稳住浙江
压住巡抚
防止皇帝疑我拖延
保住海防军心

【我的资源】
调粮权
密奏权
节制巡抚
海防军报
幕僚 4 人

【我的筹码】
半页田契暗账
县令密信
巡抚越级上奏传闻
```

## 9.3 中间：剧情消息流 + 当前决策

中间是最重要的游戏区。

中间必须像“剧情消息驱动的游戏现场”，而不是后台日志。

示例：

```text
【第 3 天 · 午后】

驿站快马离开杭州府。

有人看见浙江巡抚的幕僚亲自护送一封急奏北上。奏中称：浙江改桑已有成效，只待朝廷嘉奖，便可十日内见第一批银。

你的幕僚低声提醒：

“大人，若这封奏疏先到内阁，巡抚便是功臣；若之后民怨爆发，您这个总督就是压不住局的人。”

【你要如何应对？】

A. 截留奏疏
派人追上驿站，暂扣巡抚奏疏。
可能收益：阻止巡抚抢功。
可能风险：巡抚反咬你压制国策。

B. 追加密奏
不阻止奏疏，但另写一封密奏给皇帝。
可能收益：保留解释权。
可能风险：内阁怀疑你越级自保。

C. 放任巡抚
让他继续抢功，等待他与商会绑定更深。
可能收益：未来可一并清算。
可能风险：巡抚短期声望上升。

D. 自定义决策
输入你的处理方式。
```

## 9.4 右侧：世界反馈

右侧显示决策后变化：

```text
世界状态
人物关系
最新变化
当前风险
潜在后果
```

示例：

```text
【世界状态】
```

# 10. AI 推演设计

## 10.1 AI 不是写手，而是导演 + 裁判 + 状态机

AI 的任务不是自由写故事，而是：

```text
根据当前状态生成剧情消息
根据玩家决策判断合法性
根据决策改变状态
根据状态变化生成后续消息
控制剧情不发散
保证每天推进
最终生成结局
```

## 10.2 AI 分层

```text
AI Director：导演层
AI Judge：规则校验层
AI Narrator：叙事层
AI State Updater：状态更新层
AI Character Reactor：角色反应层
```

## 10.3 AI 角色触发条件

AI 角色不主动抢戏，只在以下情况下反应：

```text
1. 玩家影响到它
2. 主线事件推着它
3. 某个变量达到阈值
4. 它持有的把柄或风险被触发
```

例如：

```text
粮价 > 70 → 清流县令请求开仓
巡抚声望被压制 → 巡抚越级上奏
商会清算风险过高 → 商会投靠司礼监
皇帝疑心 > 75 → 司礼监入浙探查
```

# 14. 成本控制方案

AI 成本是本产品的关键风险。

必须从第一版就设计成本控制。

## 14.1 控制天数

每局固定：

```text
3 天 / 7 天 / 10 天
```

MVP：7 天。

## 14.2 控制消息数

每一天最多：

```text
3-5 条核心剧情消息
1-3 次关键决策
```

## 14.3 控制上下文长度

不要每次把全量历史都发给 AI。

应保存：

```text
session_summary
world_state
character_state
relationships
recent_messages
important_decisions
hidden_threads
```

每次只传：

```text
当前状态摘要
最近 3-5 条消息
当前待处理事件
玩家角色信息
```

## 14.4 模型分层

可采用：

```text
小模型：状态更新、合法性校验、摘要压缩
大模型：关键剧情消息、重要决策推演、最终结局
```

## 14.5 模板复用

很多剧情结构可以模板化：

```text
其他角色行动转译模板
待决策模板
结果反馈模板
日终回响模板
结局模板
```

AI 负责填充，不要每次从零生成。

## 14.6 避免无限自定义

自定义决策需要限制字数：

```text
100-300 字以内
```

并经过 AI 校验。

# 15. 关键风险与解决方案

| 风险 | 表现 | 解决方案 |
|---|---|---|
| 体验像文字游戏 | 用户觉得只是读文本 | 中间消息必须有决策压力，右侧必须有状态反馈，按钮文案要游戏化 |
| AI 发散 | 剧情越写越散 | 固定天数、每日主题、消息数量上限、状态机约束 |
| 多人卡局 | 有玩家不在线导致无法继续 | 角色默认惯性推进，AI 代为反应，不要求同步 |
| AI 成本高 | 每次推演上下文太长 | 状态摘要、消息限制、模型分层、模板化 |
| 玩家不会决策 | 选择成本高，不知道怎么选 | 每条消息给 3 个清晰选项 + 收益风险提示 |
| 自定义决策破坏世界 | 玩家乱写超能力或越权行为 | AI 合法性校验 + 改写建议 |
| 变量太多难理解 | 玩家看不懂状态 | MVP 控制在 5-7 个核心变量 |
| 平台化过早过重 | 还没验证玩法就做复杂系统 | MVP 只做一个剧本，但底层数据结构平台化 |
| 内容制作成本高 | 每个剧本都要大量手写 | 剧本模板化，角色、变量、天数、结局规则可配置 |
| 互动反馈不强 | 玩家不知道自己影响了谁 | 每次决策必须显示“最新变化”和“影响对象” |
| 不像游戏 | UI 像文档或后台 | 主页面中间必须是剧情消息流，左右是状态，不要堆配置面板 |

# 20. 最终判断

本产品最重要的不是“AI 写得多好”，而是：

> **玩家是否感觉自己真的进入了一个局，并且每一次决策都在改变局势。**

最终产品应该让用户形成这样的感受：

```text
我不是读故事。
我是在入局。
别人做的事会变成剧情压力推到我面前。
我可以压制、交易、密奏、放任或自定义出招。
我的选择会改变世界状态、人物关系和最终结局。
每一场局都有限定天数，最后必须裁决。
```

一句话总结：

> **AI 多人局不是一个单游戏，而是一个“AI 动态局势推演平台”。MVP 可以从《桑田诏》开始，但所有设计都必须服务于后续多个故事局扩展。**
