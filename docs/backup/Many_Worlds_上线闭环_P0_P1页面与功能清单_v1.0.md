# Many Worlds 上线闭环：P0 / P1 页面与功能清单

> 版本：v1.0  
> 日期：2026-07-14  
> 定位：基于当前 `aiStoryRoom` 代码的最小上线补齐范围。目标不是重做游戏，而是让用户从进入、邀请、付费到回流的关键路径不中断。

## 1. 结论与范围

当前项目的**核心游戏闭环已存在**：注册/登录、世界选择、建房或邀请码加入、选角、Ready、开始、三玩家七轮、World Credits 解锁及结果页均已有代码和本地验收基础。

本文件补的是“正常可上线产品闭环”。其中：

- **P0**：不完成就会导致用户无法继续游戏、无法完成支付、无法进入被邀请房间，或线上路由直接断开的项目。P0 全部通过才可对外开放付费和邀请。
- **P1**：提升留存、信任和售后，但不阻断第一次从首页进入并完成一局的流程。包括“我的房间 / 历史结局”等信息展示。
- **P2**：增长、运营和体验增强，首版上线不阻断。

## 2. 最小可上线用户路径

```text
首页
→ 注册 / 登录
→ 选择世界
→ 单人开始 或 创建多人房间
→ 复制邀请链接 / 被邀请人登录后入房
→ 选角、Ready、Host 开始
→ 前 3 轮免费
→ 第 4 轮余额不足时进入钱包充值
→ 外部支付
→ 支付成功确认入账
→ 自动回到原房间并完成解锁
→ 继续至第 7 轮
→ 动态结果页
→ 分享结果 / 再开一局 / 邀请下一局
```

首版原则：**复用现有页面结构，新增状态和区块优先；只有公开分享承载页需要新建。** 不改动现有主游戏 `index.html` 的既有骨架。

## 4. P0：必须闭环的页面与功能

### P0-01：线上路由、API 与部署闭环

**为什么是 P0**：本地 `apps/web/src/server.mjs` 可以处理 `/auth`、`/worlds/*`、`/rooms/*`、`/room-game`、`/game/result`，但当前 `vercel.json` 没有覆盖这些路由。用户直接刷新、从邀请链接打开或支付跳回时可能 404。

**不新增页面，只补线上路由和服务。**

| 项目 | 要求 |
|---|---|
| Web rewrite | 覆盖 `/auth`、`/worlds/:worldId`、`/rooms`、`/rooms/:roomId`、`/room-game`、`/game/result`、`/join`、`/credits-success`、`/share/:token`；刷新和直接打开均可用 |
| API | 线上同源 `/api/v4/*` 可访问；不得让生产页面依赖本地 `127.0.0.1` 代理 |
| 环境变量 | 数据库、DeepSeek、Creem、Webhook secret、公开 Web 域名、邮件发送服务均在平台密钥中配置，不能进入前端或 Git |
| 冒烟 | 线上执行首页 → 登录 → 建房 → 邀请链接 → 入房 → 一次行动 → 刷新恢复；支付只用测试环境或零金额/Mock 合同验证 |

**验收**：每条 P0 路由直接访问和刷新返回 200；线上 API 健康检查通过；浏览器 Network 中不存在对本地端口的请求。

### P0-02：房间邀请深链与登录后回房

**复用页面**：等候房继续使用 `platform.html`；邀请落地继续使用 `join.html`；无需新增视觉页面。

**当前问题**：等候房目前复制的是短邀请码；被邀请人只能手输邀请码，无法从链接自然进入，并且登录后没有稳定恢复到目标房间。

#### 用户流程

```text
Host 在房间页点击“Copy invite link”
→ 复制 https://域名/join?roomCode=XXXXXX
→ 被邀请人打开链接
→ 未登录：/auth?returnTo=/join?roomCode=XXXXXX
→ 登录 / 注册完成
→ 验证房间仍可加入
→ 调用 join-by-code
→ 进入 /rooms/:roomId
```

#### 页面信息和行为

| 页面 | 必须展示的信息 | 必须实现的操作与状态 |
|---|---|---|
| 房间等候页 | 世界名、房间名、人数/上限、邀请码、房间状态、邀请链接有效说明 | 复制链接；复制失败时展示可手工复制的链接；房满、已开始、已关闭时不可复制或明确提示 |
| `join.html` | 世界名、房间名、当前人数、邀请人昵称（可选）、“登录后加入”说明 | 已登录自动加入；未登录保存 `returnTo`；邀请码失效、房满、已开始、已关闭时给出明确去向（世界页或 Rooms） |
| `/auth` | 当前操作提示，例如“登录后加入 Night Council” | 注册/登录/验证成功后只恢复白名单内的 `returnTo`，不能开放跳转 |

#### 后端与数据

- 复用 `StoryRun.inviteCode` 和现有 `POST /v4/rooms/join-by-code`。
- 补一个仅返回安全公开信息的“邀请码预检”接口，不能泄露私密角色、隐藏目标或房间内决策。
- 需要明确邀请码状态：`active`、`revoked`、`expired`、`room_full`、`started`、`closed`。首版可先不做自定义过期时间，但至少支持 Host 撤销并禁止已开始/关闭房间加入。

**验收**：新浏览器从邀请深链开始，完成注册/登录后无需手输代码即可进入正确房间；无权限或无效链接不会暴露房间私密内容。

### P0-03：支付、成功回流与房间解锁闭环

**复用页面**：继续使用 `credits.html`、`credits-success.html`、`room-game.html`。不需要独立 Checkout 页，支付由 Creem 托管即可。

**当前问题**：已有 Checkout、Webhook 和成功页轮询，但成功页固定返回钱包；创建 Checkout 时未保存原房间和解锁意图。因此用户充值后需要自己找回房间，无法形成连续解锁。

#### 最小状态机

```text
Room paywall
→ /credits.html?returnTo=/room-game?runId=...&intent=unlock
→ 创建 checkout（服务端保存 returnTo / runId）
→ Creem
├─ 成功 → /credits-success.html?checkout_id=...
│        → 轮询 PAID → 返回 returnTo → 调用或展示“立即解锁”
├─ 取消 → 同一状态页显示“未扣款”，返回原房间
└─ 失败 / 超时 → 同一状态页显示原因、重试充值、返回原房间
```

#### 页面信息和行为

| 页面 | 必须展示的信息 | 必须实现的操作与状态 |
|---|---|---|
| `room-game.html` 解锁门槛 | 免费轮数、所需 Credits、当前可用余额、为什么只需一人解锁 | 余额足够：直接解锁；不足：跳钱包且携带 `returnTo`；解锁成功：恢复当前轮；解锁重复：不重复扣费 |
| `credits.html` 钱包/充值区 | 当前余额（bonus/purchased）、本次需补的 Credits、两个充值包、回到哪个房间 | 创建 Checkout 前禁用重复点击；失败显示可读错误；支付完成后不把用户困在钱包 |
| `credits-success.html` 通用状态页 | 支付状态、购买包、Credits 数、到账余额、原房间名称（可选） | `pending` 轮询、`paid` 自动回流、`cancelled` 返回、`failed` 重试；只允许订单所属用户查询 |
| 交易流水区（放在 `credits.html`） | 时间、类型、Credits 变化、余额变化、关联房间/订单、状态 | 分页加载现有交易 API；订单失败、退款、争议有可理解文案；不要求首版发票下载 |

#### 后端与数据

- Checkout 创建请求增加受控 `returnTo` 和可选 `runId`；服务端白名单校验后落入 Purchase metadata，**不信任前端传入的任意 URL**。
- 为订单保存 `returnTo`、`runId`、`initiatedAt`；Webhook 仍是唯一把订单改为 `PAID` 和加点的权威入口。
- 支持支付提供方的 `cancel_url` 时传入；若提供方不支持，成功状态页根据订单仍为 `PENDING` 提供“返回房间”和“继续等待”而非伪称失败。
- 复用已有 Credits ledger、`CreemPurchase`、`WorldUnlock` 的幂等约束；不得在浏览器收到跳转后直接加点。

**验收**：余额不足的用户从第 4 轮进入充值，完成测试支付后自动回到同一房间；刷新成功页和重复点击解锁均不会多加点或多扣点；取消和超时也能回原房间。

### P0-04：结果页分享与下一局回流

**复用页面**：结果页继续使用 `platform.html` 的 `/game/result`；只新建一张轻量、只读的 `share.html`（或将 `/share/:token` rewrite 到该文件）。

**当前问题**：现有结果页能“再玩一次 / 换角色 / 回世界”，但没有分享动作。旧章节 API 虽能生成 `ShareToken`，当前产品流没有调用它，也没有公开承载页。

#### 结果页必须新增的三个出口

1. **分享回顾**：生成脱敏的公开分享链接。
2. **邀请下一局**：带当前世界的建房入口；创建成功后进入 P0-02 的邀请流程。
3. **再开一局**：创建新 Run，不能复写已完成 Run。

#### 页面信息和行为

| 页面 | 必须展示的信息 | 必须实现的操作与状态 |
|---|---|---|
| `/game/result` | 世界、结局标题、关键决定、自己的角色结局、世界状态 | 分享、邀请下一局、再开一局；分享前可选择“仅公开世界结局 / 包含我的角色” |
| `/share/:token` | 世界名、公开标题、公开摘要、公开关键决定、继续探索 CTA | 只读；不显示其他角色隐藏目标、私人行动、邮箱、邀请码或完整聊天；令牌无效/撤销显示友好页 |
| 分享管理弹层（复用结果页） | 当前分享链接、公开范围、创建时间、状态 | 复制链接、撤销链接；复制失败显示文本；不做首版图片生成器 |

#### 后端与数据

- 复用 `ShareToken`，但把 `scene`、`channel`、公开范围和撤销状态明确化；Token 必须足够随机，不能由可推测 ID 拼接。
- 增加公开分享读取接口：只返回白名单字段；将“玩家私密投影”与“公开世界回顾”分开构建。
- 增加撤销接口，仅分享创建者或房主可调用；撤销后公开链接立即失效。
- 首版无需生成海报图片、排行榜或社媒自动发布；页面 title、description 和分享缩略图可使用固定品牌图。

**验收**：完成一局后可生成链接；无登录浏览器可看到脱敏回顾；撤销后返回不可用；任何分享接口都不能读到私密行动或隐藏目标。

---

### P0-05：生产安全边界与统一异常落点

**复用页面**：在 `platform.html`、`credits-success.html`、`join.html` 增加状态区块，不额外做设计稿。

| 场景 | 页面表现 | 必须动作 |
|---|---|---|
| 会话过期 | “登录已过期” + 继续按钮 | 保存受控 `returnTo`，登录后恢复 |
| 房间不存在/满员/已开始/已关闭 | 明确原因 + 去世界页/Rooms | 不展示房间私密数据 |
| 订单不存在/非本人订单 | 明确“无法读取此订单” | 不泄露订单金额、邮箱、余额 |
| Webhook 延迟 | “付款已收到，正在到账” + 继续等待/返回房间 | 不伪造支付成功；可安全刷新 |
| API/网络异常 | 不显示技术堆栈 | 可重试且避免重复创建房间、重复 Checkout、重复解锁 |

生产环境不得展示“Create local test account”或把邮箱验证 token 回显给浏览器；测试邮箱、测试加点和 Mock Checkout 必须受环境开关约束。

## 5. P1：上线后紧接着补齐的信息与留存功能

### P1-01：我的房间与历史结局（复用 `/rooms`）

这是信息展示和恢复能力，不阻断首局，所以列为 P1。

| 区块 | 展示信息 | 操作 |
|---|---|---|
| 进行中 | 世界、房间名、角色、当前轮次、最后更新时间、是否轮到我 | Continue |
| 等候中 | 房间人数、Ready 状态、邀请码状态 | Open room / Copy invite |
| 已完成 | 世界、结局标题、完成时间、我的角色 | View result / Play again / Share |
| 已关闭/失效 | 原因、关闭时间 | 从列表移除或只读展示 |

要求使用真实 `myRooms` 数据，不保留给用户可点击的 fixture 占位房间。

### P1-02：账户中心（复用 `credits.html` 的账户区）

首版不新建 `/account`，而是在钱包页添加 tabs：`账户`、`World Credits`、`订单与流水`。

| Tab | 信息 | 操作 |
|---|---|---|
| 账户 | 昵称、邮箱、验证状态、注册时间 | 修改昵称、重发验证、改密码、退出 |
| Credits | bonus/purchased/debt、即将过期 bonus | 充值、查看流水 |
| 订单与流水 | 订单号短码、包、金额、状态、时间 | 查看详情、联系支持 |

账户删除和数据导出至少提供“联系支持申请”的入口；如在目标市场有强制自助要求，再升级为 P0。

### P1-03：退款、支持与状态页（复用 `legal.html`）

- 退款政策页增加“联系支持”按钮和订单号输入说明。
- 支持入口必须包含：订单/房间 ID、问题分类、邮箱、描述；可先邮件表单或 `mailto:`，无需首版工单系统。
- 增加公开状态地址或至少在帮助页说明维护时的状态反馈方式。

### P1-04：邀请与分享转化信息

- 钱包页显示推荐人数、已完成开局人数、已奖励数、剩余奖励名额。
- 记录真实分享渠道（copy/native/X/Facebook），但奖励仍只在新用户完成开局后发放。
- 不做首版排行榜、裂变排行榜或强制社交登录。

## 6. P2：不阻断上线的增强项

- 分享海报、图片下载、社媒一键发布、Open Graph 动态图片。
- 收藏世界、搜索/筛选世界、创作者页、公司/招聘等营销页。
- 通知中心未读数、邮件通知偏好、推送通知。
- 完整客服工单后台、发票下载、订阅制、优惠码、地区税务。
- 复杂账号安全：多设备会话列表、二次验证、完整数据导出自助流程。

## 7. 最小 API / 数据补齐清单

| 能力 | 最小接口或字段 | 优先级 |
|---|---|---|
| 邀请预检 | `GET /v4/rooms/invite/:code`，只返回公开房间摘要和可加入状态 | P0 |
| Checkout 回流 | `POST /v4/billing/checkouts` 接收受控 `returnTo` / `runId`；Purchase 保存该上下文 | P0 |
| 支付状态 | 现有 `GET /v4/billing/checkouts/:checkoutId` 返回 `returnTo` 所需的安全摘要 | P0 |
| 交易流水 | 复用 `GET /v4/credits/transactions`，补关联订单/房间展示字段 | P0 |
| 分享创建/读取/撤销 | `POST /v4/runs/:runId/share`、`GET /v4/public/share/:token`、`DELETE /v4/shares/:token` | P0 |
| 房间邀请状态 | Invite 状态或撤销时间；禁止已开始/关闭房间的新加入 | P0 |
| 账户操作 | 更新昵称、退出（客户端清 token）、重发验证、修改密码 | P1 |
| 我的房间/结局 | 现有 Rooms 查询返回真实 `myRooms` 和可继续动作 | P1 |

## 8. P0 验收门禁

只有以下项目全部通过，才将产品称为“可上线闭环”：

1. 新用户可从首页完成单人或多人第一局，不依赖开发者手输 ID 或邀请码。
2. 邀请链接在无登录浏览器中打开，登录/注册后自动加入正确房间；所有无效状态有明确反馈。
3. 第 4 轮余额不足时，用户能充值、确认到账、回到同一房间并完成一次且仅一次解锁。
4. 支付取消、支付失败、Webhook 延迟、刷新支付成功页均不会丢失用户去向或重复扣款。
5. 每局结果页都能生成脱敏分享链接；公开链接不可泄露私密行动、隐藏目标、邮箱或邀请码；撤销立即生效。
6. 上述路由在生产域名上直接打开和刷新均正常，线上 API、数据库、Webhook、邮件和环境变量配置经独立验证。
7. 保留既有三玩家七轮、Credits 账本幂等、权限投影和结果页回归，不因补产品流程破坏主游戏。

## 9. 建议实施顺序

1. P0-01 先补生产路由/API 部署合同，否则邀请、支付回跳、分享链接都没有可靠落点。
2. P0-02 邀请深链与 `returnTo`，先让多人拉新能进入游戏。
3. P0-03 支付回流和交易流水，打通付费继续游戏。
4. P0-04 结果分享，完成“玩完 → 分享/下一局”的增长回路。
5. P0-05 统一异常与生产安全检查，然后做一轮线上端到端验收。
6. 再做 P1 的我的房间、历史结局、账户和支持入口。

## 10. 明确不做，避免拖慢首版上线

- 不重做主游戏视觉和已通过的七轮玩法。
- 不做内部支付表单；继续使用 Creem 托管 Checkout。
- 不做订阅、优惠码、积分商城、复杂发票系统。
- 不做多层级裂变、排行榜或自动发布社媒。
- 不做完整后台客服系统；P1 先有可追踪的联系支持入口即可。
