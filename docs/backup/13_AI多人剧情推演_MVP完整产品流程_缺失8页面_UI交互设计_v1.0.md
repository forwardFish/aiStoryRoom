# AI 多人剧情推演：完整产品流程缺失页面 UI / 交互设计文档 v1.0（MVP）

> v1.0 工程对齐说明：当前主分支正式页面主要为 `/`、`/role-select`、`/game`；本文件补充登录注册、剧本详情、房间大厅、创建房间、加入房间、房间等待与准备、我的游戏、游戏结局与本局总结共 8 个独立页面。
>
> 本文档与《AI 多人剧情推演：主游戏页持续叙事流 UI / 交互设计文档 v1.1》配套使用：
>
> - 本文档负责进入主游戏之前和结束之后的完整产品流程。
> - 原主游戏文档负责 `/game?runId=<runId>` 内部的 8 个叙事状态。
> - 两份文档共同构成“首页 → 剧本 → 登录 → 房间 → 角色 → 游戏 → 结局”的完整 MVP 页面标准。
>
> 文档用途：
>
> 1. 作为 Codex 开发新增 8 个独立页面的产品与交互规格。
> 2. 作为 ChatGPT / UI 设计工具生成 8 张独立设计图的统一依据。
> 3. 作为前端、后端、自动化测试和最终验收的页面标准。
>
> 视觉基准：
>
> - 以当前 Many Worlds 首页、角色选择页和新版主游戏页为统一视觉基准。
> - 使用大面积白色与浅灰紫背景、紫蓝品牌色、淡蓝紫水墨山水、细描边、轻阴影和适度圆角。
> - 不继续使用旧版重米黄宣纸、朱红主按钮和过度古典化界面。
> - 历史剧本的时代感主要由水墨背景、人物立绘、标题字体、纹样和内容建立；平台框架仍保持现代、开放和多世界感。

## 1. 产品流程的核心定义

当前产品不是缺少 AI 推演能力，而是缺少把用户稳定送入推演、让多人在同一个房间完成准备、并在结局后形成完整闭环的产品流程。

完整体验应当是：

```text
发现一个世界
  ↓
了解剧本和玩法
  ↓
登录并保留进入意图
  ↓
选择单人或多人
  ↓
创建房间或加入房间
  ↓
多人选角色并准备
  ↓
进入主游戏持续推演
  ↓
完成最终结局
  ↓
回看个人命运与多人影响
  ↓
重新开始或进入其他世界
```

八个新增页面的分工：

```text
P0-01 登录 / 注册
解决“你是谁、如何恢复账号与原进入路径”

P0-02 剧本详情
解决“这个世界是什么、是否值得进入、玩哪种模式”

P0-03 房间大厅
解决“有哪些真实房间、我已经参与了哪些局”

P0-04 创建房间
解决“我要用什么规则开启一个新多人局”

P0-05 加入房间
解决“我如何通过邀请码或链接进入别人的局”

P0-06 房间等待与准备
解决“谁在房间、谁选了什么角色、何时可以开始”

P0-07 我的游戏
解决“我回来以后从哪里继续之前的局”

P0-08 游戏结局与本局总结
解决“这一局最终发生了什么、我改变了谁、下一步做什么”
```

最重要的产品原则：

> 首页负责发现剧本，剧本详情负责做进入决策，房间大厅负责选择具体房间，等待房间负责多人准备，主游戏负责推演，结局页负责完成体验。任何一个页面都不应承担其他页面的全部职责。

## 2. 页面实现方式

### 2.1 设计稿数量

MVP 新增 **8 张独立 UI 图**：

| 编号 | 页面名称 | 建议路由 | 页面类型 |
|---|---|---|---|
| P0-01 | 登录 / 注册 | `/auth` | 独立路由 |
| P0-02 | 剧本详情 | `/stories/:storyId` | 独立路由 |
| P0-03 | 房间大厅 | `/rooms` | 独立路由 |
| P0-04 | 创建房间 | `/rooms/create?storyId=<storyId>` | 独立路由 |
| P0-05 | 加入房间 | `/join?code=<inviteCode>` | 独立路由 |
| P0-06 | 房间等待与准备 | `/rooms/:roomId` | 独立路由 |
| P0-07 | 我的游戏 | `/me/games` | 独立路由 |
| P0-08 | 游戏结局与本局总结 | `/game/result?runId=<runId>` | 独立路由 |

### 2.2 与现有页面的关系

当前已有：

```text
/                 首页 / 剧本目录
/role-select      单人角色选择
/game             主游戏
```

调整后建议：

```text
首页
  ↓
剧本详情
  ↓
登录校验
  ↓
模式选择
  ├─ 单人 → /role-select → /game
  └─ 多人 → 创建/加入房间 → 等待房间选角色 → /game
  ↓
/game/result
```

### 2.3 单人与多人角色选择必须分开

单人模式：

```text
/stories/sangtian
  ↓ 选择单人模式
/role-select?story=sangtian
  ↓ 创建单人 Run
/game?runId=<runId>
```

多人模式：

```text
/stories/sangtian
  ↓ 创建或加入多人房间
/rooms/:roomId
  ↓ 在等待房间中选择未被占用角色
  ↓ 全员准备并开始
/game?runId=<runId>&roomId=<roomId>
```

多人模式不应先进入现有单人 `/role-select` 页面，否则无法正确处理角色占用、玩家准备和房主开始游戏。

```text

```

### 3.4 水墨背景使用规则

- 水墨只作为弱背景，不压过正文和控件。
- 水墨透明度建议 6%—14%。
- 历史剧本可以使用山水、城池、府衙、河道。
- 科幻、都市、奇幻剧本后续可切换为相应世界背景。
- 平台外壳的结构、按钮、卡片不因剧本改变。

## 4. 全局认证与返回规则

### 4.1 必须登录的页面

```text
/rooms
/rooms/create
/join 的最终确认加入动作
/rooms/:roomId
/me/games
/game
/game/result
```

### 4.2 可以未登录浏览的页面

```text
/
 /stories/:storyId
/auth
```

### 4.3 登录后返回原路径

用户从以下页面触发登录：

```text
/stories/sangtian
/join?code=ABCD12
/rooms/create?storyId=sangtian
/game?runId=xxx
```

登录页必须带：

```text
/auth?returnTo=<encoded-url>
```

登录成功后返回 `returnTo`，不能统一跳首页。

### 4.4 登录失效

在受保护页面请求出现 `401` 时：

```text
保存当前 URL
清除本地无效会话
跳转 /auth?returnTo=<current-url>&reason=session_expired
```

登录页显示：

```text
登录状态已过期，请重新登录。
完成后会回到刚才的页面。
```

# 5. P0-01：登录 / 注册页

## 5.1 使用场景

- 用户准备创建单人局。
- 用户准备创建或加入多人房间。
- 用户访问“我的游戏”。
- 用户访问已有 Run，但登录状态失效。
- 用户从邀请链接进入，但尚未登录。
- 新用户领取初始 World Credits。

## 5.2 页面目标

让用户在最少步骤内完成认证，同时明确：

- 登录后会回到哪里。
- 注册后能获得什么。
- 当前操作不会丢失。
- 登录和注册属于进入世界前的必要步骤，而不是独立后台流程。

## 5.3 页面结构

```text
[顶部 Logo]                                   [返回发现世界]

[左侧品牌叙事区]              [右侧认证卡片]

世界不会暂停，
你的身份会被世界记住。

每一次选择、关系变化和最终结局，
都会保存在你的 Many Worlds 账户中。

[淡水墨多世界插画]

                               登录 / 注册切换
                               邮箱
                               密码
                               忘记密码
                               登录按钮

                               或

                               Google 登录（后续可选）
                               注册新账户
```

桌面比例建议：

```text
左侧 52%
右侧 48%
认证卡片宽度 440px—480px
```

## 5.4 登录状态

标题：

```text
欢迎回到 Many Worlds
```

说明：

```text
登录后继续进入《嘉靖财政危局》。
```

字段：

```text
邮箱地址
密码
记住我
忘记密码
```

主按钮：

```text
登录并继续
```

次级入口：

```text
还没有账户？免费注册
```

## 5.5 注册状态

标题：

```text
创建你的 Many Worlds 身份
```

字段：

```text
邮箱地址
设置密码
确认密码
同意用户协议和隐私政策
```

权益提示：

```text
注册后获得：
· 保存所有故事局进度
· 加入朋友的多人世界
· 领取新用户 World Credits
```

主按钮：

```text
注册并继续
```

## 5.6 忘记密码状态

可以使用同页卡片切换，不增加新设计稿。

内容：

```text
找回密码

输入注册邮箱，我们会发送重置链接。

[邮箱输入框]

发送重置邮件
返回登录
```

发送成功：

```text
邮件已发送

请检查 your@email.com。
若几分钟后仍未收到，请检查垃圾邮件。
```

## 5.7 邀请链接进入时

如果 `returnTo=/join?code=ABCD12`，卡片顶部显示轻量提示：

```text
你正在加入朋友创建的故事局

房间：嘉靖财政危局 · 江南夜谈
邀请码：ABCD12
```

不能提前展示其他玩家隐私数据。

## 5.8 错误状态

```text
邮箱或密码错误
邮箱格式不正确
密码至少 8 位
该邮箱已经注册
登录请求超时
会话已过期
```

错误显示在对应字段下方，不使用浏览器原生弹窗。

## 5.9 交互要求

- Enter 提交当前表单。
- 提交期间按钮显示“登录中……”或“注册中……”。
- 防止重复提交。
- 密码支持显示 / 隐藏。
- 注册成功后自动登录。
- 认证成功后优先跳转 `returnTo`。
- 没有 `returnTo` 时跳转 `/rooms` 或 `/me/games`，不要跳到 Credits 页面。

## 5.10 页面不应出现

- 复杂账户资料填写。
- 用户昵称、头像、性别、生日等非必要字段。
- 要求用户先充值。
- 大段技术安全说明。
- 单独的管理员登录入口。

## 5.11 设计图必须表现

- 登录状态，不必同时画注册表单。
- 左侧品牌世界感。
- 右侧清晰认证卡片。
- “登录后继续进入《嘉靖财政危局》”返回意图提示。
- Many Worlds 紫蓝视觉体系。

# 6. P0-02：剧本详情页

## 6.1 使用场景

用户从首页的：

```text
立即入局
查看详情
剧本卡片
分享链接
```

进入某个剧本。

## 6.2 页面目标

在进入角色选择或房间之前，让用户明确：

- 这个世界讲什么。
- 我要扮演谁。
- 游戏大约多久。
- 单人和多人有什么区别。
- 是否需要 World Credits。
- 下一步是单人开始还是创建 / 加入多人房间。

## 6.3 页面结构

```text
[全局顶部导航]

[剧本 Hero]
左：封面 / 水墨城池
中：标题、简介、标签、元信息
右：进入操作卡

[世界背景]
[核心冲突]
[可选角色]
[玩法特点]
[单人 / 多人模式对比]
[本局消耗与规则]
[底部进入操作]
```

## 6.4 Hero 区域

示例：

```text
权谋历史 · AI 动态推演

桑田诏：嘉靖财政危局

嘉靖朝财政濒临崩溃。
七天之内，你必须在皇权、粮价、民心、海防和官场暗斗之间作出选择。

7 天
1—5 人
30—45 分钟 / 人
多结局
持续叙事
角色视角
```

主要按钮：

```text
开始单人故事
创建多人房间
```

次级入口：

```text
输入邀请码加入
```

## 6.5 进入操作卡

```text
选择游玩方式

单人故事
· AI 扮演其他角色
· 随时继续
· 适合首次体验
[开始单人故事]

多人共演
· 朋友分别扮演不同角色
· 每个人拥有独立秘密和目标
· 行动会进入彼此故事
[创建多人房间]

已有邀请码？
[加入房间]
```

## 6.6 世界背景

不使用长篇百科介绍，控制在 3—5 段。

示例：

```text
国库空虚，东南海防欠饷。

朝廷要求浙江大规模改稻为桑，用丝绸换取白银。
然而粮田一旦减少，粮价与民乱将首先冲击江南。

总督、巡抚、县令、商会和织造使表面都在执行国策，
实际上每个人都在保护自己的仕途、财富或性命。
```

## 6.7 核心冲突

使用三到四张轻量信息卡：

```text
皇命与地方现实
财政与民生
合作与背叛
短期收益与延迟后果
```

每张只写一句解释，不预测具体结局。

## 6.8 可选角色

横向角色卡：

```text
浙江总督
统筹全局 · 高权力 · 中风险

浙江巡抚
争功晋进 · 中权力 · 高风险

清流县令
清流查账 · 证据强 · 高风险

江南商会
逐利观望 · 资源多 · 中风险

司礼监织造使
暗中监视 · 情报强 · 高风险
```

点击角色只打开轻量详情，不直接选择角色。

详情内容：

```text
公开身份
公开目标
命运问题
玩法特点
```

不展示隐藏目标和隐藏资源。

## 6.9 单人 / 多人对比

| 项目 | 单人故事 | 多人共演 |
|---|---|---|
| 其他角色 | AI 控制 | 真人玩家控制 |
| 节奏 | 随时推进 | 等待同轮玩家 |
| 角色冲突 | AI 动态生成 | 真人选择互相影响 |
| 推荐人群 | 初次体验 | 朋友共同游玩 |
| 进入方式 | 选择角色后开始 | 创建或加入房间 |

## 6.10 Credits 显示

示例：

```text
本剧本消耗

单人：10 World Credits / 局
多人：每位玩家 10 World Credits / 局

开始游戏时扣除。
房间未成功开始不扣除。
```

如果免费测试：

```text
MVP 测试期间免费
原价 10 World Credits
```

## 6.11 交互要求

- “查看详情”必须进入本页，不能继续直接跳角色选择。
- “开始单人故事”登录后进入 `/role-select?story=sangtian`。
- “创建多人房间”登录后进入 `/rooms/create?storyId=sangtian`。
- “加入房间”进入 `/join`，并预填剧本信息。
- 登录前点击任何需要登录的按钮，跳 `/auth?returnTo=...`。
- 如果用户已有该剧本进行中的单人局，操作卡顶部显示：

```text
你有一个进行中的故事
第 3 天 · 浙江总督
[继续故事]
[重新开始]
```

## 6.12 页面不应出现

- 房间列表。
- 完整玩家社交动态。
- 所有结局剧透。
- 复杂数值系统说明。
- 隐藏目标。
- 直接创建固定 Run。

## 6.13 设计图必须表现

- 剧本 Hero。
- 单人 / 多人主要入口。
- 5 个角色概览。
- 世界背景和核心冲突。
- Credits 消耗说明。
- 统一浅紫水墨视觉。

# 7. P0-03：房间大厅页

## 7.1 使用场景

- 登录后主动查看可加入房间。
- 从导航点击“房间大厅”。
- 从首页想进入真实多人局。
- 用户想继续自己已经参与的等待房间或进行中房间。

## 7.2 页面目标

明确区分：

```text
剧本：玩哪个世界
房间：和哪些具体玩家一起玩
```

让用户可以：

- 快速继续自己的房间。
- 创建新房间。
- 通过邀请码加入。
- 浏览公开等待房间。
- 识别房间是否已经开始、是否可加入。

## 7.3 页面结构

```text
[全局顶部导航]

[页面标题]
多人故事房间
和朋友进入同一个世界，分别成为其中的主角。

[主操作]
创建房间
输入邀请码

[我的房间]
```

# 8. P0-04：创建房间页

## 8.1 使用场景

- 从剧本详情点击“创建多人房间”。
- 从房间大厅点击“创建房间”。
- 从结局页点击“邀请朋友再开一局”。

## 8.2 页面目标

用尽量少的配置创建一个可邀请、可等待、可开始的多人局。

MVP 不是复杂服务器管理页，默认值应当足够合理。

## 8.3 页面结构

```text
[全局顶部导航]

创建多人故事局

[左侧配置表单]                [右侧房间预览与费用]

选择剧本
房间名称
玩家人数
公开范围
决策等待时间
是否允许中途加入

                              剧本封面
                              房间摘要
                              Credits 消耗
                              [创建房间]
```

桌面比例：

```text
左侧 64%
右侧 36%
```

## 8.4 基础信息

### 剧本

从剧本详情进入时锁定：

```text
桑田诏：嘉靖财政危局
7 天 · 3—5 人 · 权谋历史
```

允许点击“更换剧本”返回剧本选择，不在本页展开完整剧本库。

### 房间名称

默认：

```text
长安客的嘉靖财政危局
```

限制：

```text
2—30 个字符
```

### 玩家人数

根据剧本限制：

```text
3 人
4 人
5 人
```

默认推荐人数，例如 3 人 MVP。

## 8.5 房间可见性

```text
仅邀请可加入（默认）
拥有邀请码或邀请链接的用户可加入。

公开可加入
房间会显示在房间大厅中。
```

公开房间可以增加：

```text
直接加入
房主审核后加入
```

MVP 可以先只支持“直接加入”。

## 8.6 决策等待时间

```text
不限时（推荐朋友异步游玩）
30 分钟
2 小时
12 小时
24 小时
```

说明：

```text
超过等待时间后，房主可以选择继续等待、由 AI 托管或暂停房间。
```

MVP 若尚未实现 AI 托管，只显示：

```text
超过等待时间后，房主可以暂停或继续等待。
```

## 8.7 中途加入

```text
游戏开始后不允许新玩家加入（默认）
```

MVP 建议固定关闭，不提供可切换选项，避免角色与上下文一致性问题。

## 8.8 房主预选角色

可选：

```text
稍后在等待房间选择（默认）
预选浙江总督
```

多人角色最终仍需在等待房间确认。

## 8.9 Credits 费用

右侧预览：

```text
本房间预计 3 位玩家

每位玩家：10 World Credits
房主当前余额：36 World Credits

房间创建不扣费。
全员开始游戏时分别扣除。
```

余额不足不阻止创建房间，但开始游戏前必须处理。

## 8.10 创建确认

主按钮：

```text
创建房间
```

提交中：

```text
正在创建……
```

成功：

```text
创建成功
→ /rooms/:roomId
```

必须生成：

- `roomId`
- `inviteCode`
- 邀请链接
- 房主成员记录
- 房间剧本与人数配置

## 8.11 错误状态

```text
房间名称为空
玩家人数不符合剧本规则
剧本不可创建多人局
创建请求超时
重复点击导致的重复创建
```

创建接口必须使用幂等键，避免产生两个房间。

## 8.12 页面不应出现

- 复杂权限矩阵。
- 自定义 AI 模型。
- Temperature、Token、上下文等技术选项。
- 主游戏决策规则配置。
- 可编辑角色隐藏目标。
- 游戏商城式增值选项。

## 8.13 设计图必须表现

- 左侧清晰表单。
- 右侧固定房间预览。
- “创建不扣费，开始时扣费”提示。
- 剧本已经预选为《嘉靖财政危局》。
- 主按钮清晰但不过度占满全页。

# 9. P0-05：加入房间页

## 9.1 使用场景

## 9.2 页面目标

让用户在确认加入之前清楚知道：

- 邀请码是否有效。
- 这是哪个剧本和房间。
- 谁创建了房间。
- 当前有多少玩家。
- 还剩哪些角色。
- 是否需要 Credits。

```text
加入朋友的故事局

邀请码通常由房主通过链接或消息发送。
```

## 9.4 查找到房间后的结构

```text
[左侧剧本与房间信息]           [右侧加入确认]
```

右侧：

```text

```

## 9.5 未登录状态

仍可展示有限房间信息：

```text

2 / 5 人
```

主按钮：

```text

```

跳转：

```text

```

## 9.6 Credits 不足

```text
加入房间不立即扣费。
开始游戏时需要 10 World Credits。

当前余额：4
```

按钮仍可为：

```text

```

同时提示：

```text

```

避免用户因为暂时余额不足而无法进入朋友等待房间。

## 9.7 异常状态

### 邀请码无效

```text

```

### 房间已开始

```text

```

如果用户本来就在成员列表：

```text

[返回房间]
```

### 房间已解散


## 9.8 交互要求

- 输入邀请码后自动转换为大写。
- 支持一次性粘贴。
- 有 `code` 参数时自动查询，不要求再次点击。
- 查询中不清空邀请码。
- 加入请求必须防重复。
- 加入成功跳 `/rooms/:roomId`。

- 其他玩家 Credits 余额。

- 游戏结果。
- 强制先购买 Credits。
- 未验证邀请码就展示完整房间数据。

## 9.10 设计图必须表现

- 已查找到有效邀请码后的确认状态。
- 剧本封面与房间信息。
- 当前玩家头像。
- 剩余角色数量。
- 加入后选角色的说明。

```text
· 取消
```

## 10.13 异常状态


## 10.14 页面不应出现

- 主游戏故事正文。
- 玩家隐藏任务。
- 其他玩家私人资源。
- 复杂语音、聊天、好友系统。
- 大量管理员设置。
- 可以重复选择同一角色的前端假状态。

## 10.15 设计图必须表现

- 3 位已加入玩家和 2 个空位。
- 中间 5 张角色卡。
- 至少一个角色已被他人占用。
- 当前用户已选择角色但尚未准备。
- 右侧开始条件。
- 房主按钮处于“还有玩家未准备”的禁用状态。
- 邀请码和复制链接入口。

## 11.8 空状态

完全没有游戏：


```text

```

## 11.9 交互要求

根据状态动态决定主按钮：

```text
waiting_room       → 回到房间
needs_role         → 选择角色
needs_ready        → 去准备
needs_decision     → 继续决策
waiting_others     → 查看故事
in_progress        → 继续游戏
completed          → 查看结局
dissolved          → 查看记录（可选）
```

- 页面进入后拉取用户所有 membership 与 run。
- 不依赖单一 localStorage Run ID。
- 相同 Room / Run 不重复显示。
- 已完成多人房间按玩家自己的角色显示个人结局摘要。
- 数据加载失败允许重新加载。

- 公开房间推荐。
- 复杂账户设置。
- 其他玩家隐藏信息。
- 只显示一个“继续上次游戏”而看不到其他局。
- 依赖浏览器本地记录作为唯一来源。

## 11.11 设计图必须表现

- 顶部四个状态数字。
- 至少四张卡：待我决策、等待开始、等待他人、已完成。
- 主操作随状态变化。
- 视觉上“待我决策”权重最高。
- 支持单人和多人角标。

# 12. P0-08：游戏结局与本局总结页

## 12.1 使用场景

- 单人第 7 天完成。
- 多人房间完成最终推演。
- 用户从“我的游戏”查看已完成故事。
- 用户通过分享链接查看允许公开的结局卡。

## 12.2 页面目标

让用户清楚感受到：

```text
这一局结束了
我的选择产生了真实后果
我的角色拥有完整命运
别人改变了我，我也改变了别人
这不是简单的 AI 文本停止生成
```

## 12.3 页面结构

```text
[沉浸式结局 Hero]

最终结局标题
世界结局一句话
玩家角色命运一句话

[本局世界状态]
[我的角色结局]
[关键选择]
[多人影响]
[最终故事]
[分享与再次游玩]
```

顶部仍保留 Many Worlds Logo 和基础导航，但可以弱化普通导航，提升结局仪式感。

## 12.4 结局 Hero

示例：

```text
第七日 · 御前裁决

江南未乱，朝局已变

你稳住了杭州粮价，也保住了海防军饷。
但你交给皇帝的奏疏，让巡抚成为新的政敌。

浙江总督 · 郑坤彬
结局：守土者，亦是新局中的孤臣
```

背景：

- 淡蓝紫江南山水。
- 天光比主游戏稍明亮。
- 使用结局专属印章或章节纹样。
- 不做黑屏结算。

## 12.5 世界最终状态

用 4—6 个指标和一句解释：

```text
国库银        38 万两
民心          61
粮价          66
改革进度      74%
皇帝信任      49
海防稳定      72
```

下方总结：

```text
改桑没有完全停止，但粮田得到保留。
商会被迫放粮，朝廷获得短期白银，江南暂时没有爆发民乱。
```

不展示内部评分公式。

## 12.6 我的角色结局

```text
你的公开目标

✓ 稳定浙江局势
✓ 避免民乱与海防失饷
✕ 没能阻止巡抚坐大

完成 2 / 3
```

命运问题回响：

```text
你是在保浙江，还是在保自己的官位？

最终答案：
你保住了浙江，却再也无法只做一个地方官。
```

这部分应当比普通“任务完成率”更有故事感。

## 12.7 关键选择回顾

最多展示 5 个最重要选择：

```text
第 1 天
你扣下巡抚奏疏，第一次改变了朝廷看到事件的顺序。

第 3 天
你选择向商会承诺暂缓税赋，换取粮仓开放。

第 5 天
你没有公开织造使的密信，让其影响在两日后反噬。
```

每条只显示：

- 当时的决定。
- 后续造成的可见影响。
- 不展示模型推理和隐藏权重。

## 12.8 多人影响

多人局必须有：

```text
你改变了谁

· 清流县令因你的密奏免于成为替罪羊。
· 江南商会因你的承诺提前开放粮仓。

谁改变了你

· 巡抚提交的奏疏让皇帝对你产生怀疑。
· 织造使隐藏的密报迫使你改变最终证词。
```

仍遵守视角限制：

- 结局阶段可以公开已经在故事中揭示的行动。
- 未被揭示的私人目标仍不直接公布。
- 可以显示“某个未查明的行动”而不是强行揭底。

## 12.9 个人称号

示例：

```text
你的主角类型

守局者
善于控制局势、延缓危机，
但会把更大的冲突留到未来。
```

用于分享卡，不参与强排名。

## 12.10 最终故事

入口：

```text
阅读最终故事
```

展开后按照 POV 分章：

```text
第一章：浙江总督
第二章：清流县令
第三章：江南商会
第四章：御前奏疏
尾声：江南未乱
```

MVP 可以先在同页展开，不增加独立小说路由。

## 12.11 分享卡

```text
生成我的结局卡
```

卡片包含：

- Many Worlds Logo
- 剧本名
- 角色名
- 结局标题
- 主角类型
- 关键数据 2—3 项
- 邀请朋友进入链接

不包含：

- 其他玩家隐私。
- 完整剧透。
- 用户邮箱。
- 隐藏目标。

## 12.12 再次游玩

主操作：

```text
更换角色再玩一次
```

多人局：

```text
邀请原房间成员再开一局
```

次级操作：

```text
返回我的游戏
发现其他世界
```

重新开始必须创建新 Run，不能覆盖原结局。

## 12.13 异常与生成状态

结局生成中：

```text
世界正在收束所有人的命运……

正在生成最终结局与个人故事。
```

不展示模型步骤。

部分内容生成失败：

```text
结局已保存
最终故事暂时未生成完成

[重新生成最终故事]
```

不得因为小说生成失败丢失 Run 完成状态。

## 12.14 页面不应出现

- 技术评分过程。
- Token 消耗。
- AI 模型名称。
- 全部隐藏目标和私密行动强制公开。
- 单纯“游戏完成”加一个返回按钮。
- 竞争式排行榜作为主要结局反馈。

## 12.15 设计图必须表现

- 完整结局 Hero。
- 世界最终状态。
- 角色目标完成情况。
- 关键选择回顾。
- 多人影响摘要。
- “生成结局卡”“更换角色再玩一次”“返回我的游戏”。

## 18. 加载、错误与幂等

### 18.1 创建房间

必须带：

```text
idempotencyKey
```

重复点击只返回同一个房间。

### 18.2 加入房间

同一用户重复加入同一房间：

```text
返回已有 membership
```

不能新增重复成员。

### 18.3 选择角色

角色锁定必须由服务端原子完成：

```text
roomId + roleKey 唯一占用
```

冲突返回：

```text
ROLE_ALREADY_TAKEN
```

前端提示：

```text
这个角色刚刚被其他玩家选择，请选择另一个角色。
```

### 18.4 开始游戏

多人开始必须是事务：

```text
验证房主
验证房间状态
验证人数
验证全部角色唯一
验证全部已准备
验证全部 Credits
统一扣除 Credits
创建多人 Run
关联全部成员角色
更新房间为 in_progress
发布 room.started
```

任何一步失败都整体回滚。

### 18.5 结局生成

Run 完成与最终小说生成分开保存：

```text
Run 可以先 completed
FinalStory 可以异步生成
```

小说生成失败不影响用户查看角色结局和世界状态。

## 19. 视觉设计规范

### 19.1 总体风格

必须与当前新版 Many Worlds 保持一致：

- 白色和浅灰紫页面。
- 紫蓝渐变作为品牌主色。
- 淡蓝紫水墨世界背景。
- 深蓝黑标题与正文。
- 细线描边和轻阴影。
- 卡片圆角适中。
- 历史角色立绘低饱和、写实国风。
- 控件仍然现代、清晰、可交互。

### 19.2 不允许出现

- 八个页面各自使用完全不同风格。
- 旧式全米黄宣纸后台。
- 大面积朱红按钮。
- 黑色全屏游戏大厅。
- 霓虹赛博风。
- 复杂玻璃拟态。
- Steam 商城式卡片堆积。
- 手机 App 样式直接拉伸到桌面。
- 大量渐变文字。
- 过度花哨的古典边框。

### 19.3 主按钮

默认：

```text
紫蓝渐变
白色文字
高度 48px—54px
圆角 10px—12px
```

禁用：

```text
浅灰紫背景
低对比文字
无发光阴影
```

危险操作：

```text
仅在确认弹窗中使用红色文字或红色按钮
```

### 19.4 卡片层级

```text
一级页面卡片：白色、1px 边框、轻阴影
二级信息块：浅紫或浅灰背景、无强阴影
标签：低饱和背景、深色文字
```

## 23. 开发验收标准

### 23.1 完整流程

- [ ] 首页点击剧本可以进入剧本详情。
- [ ] 剧本详情可以选择单人或多人。
- [ ] 未登录用户会进入登录页。
- [ ] 登录成功回到原进入路径。
- [ ] 单人可以进入现有角色选择并创建 Run。
- [ ] 多人可以创建或加入真实房间。
- [ ] 多人可以在等待房间选角色和准备。
- [ ] 全员准备后房主可以开始。
- [ ] 完成游戏后进入结局页。
- [ ] 用户可以从“我的游戏”恢复任意进行中游戏。

### 23.2 登录

- [ ] 注册和登录都能正常提交。
- [ ] 登录失败有字段级错误。
- [ ] 会话失效后能返回原 Run。
- [ ] 邀请链接登录后不会丢失邀请码。
- [ ] 主游戏入口必须有登录校验。

### 23.3 剧本详情

- [ ] “查看详情”不再与“立即入局”执行同一跳转。

- [ ] 显示 Credits 规则。

### 23.4 房间大厅

- [ ] 我的房间和公开房间分开展示。

- [ ] 房间状态和按钮一致。
- [ ] 可以创建房间和输入邀请码。
- [ ] 已开始和已满房间不能错误加入。

### 23.5 创建与加入

- [ ] 创建房间防重复。
- [ ] 加入房间防重复成员。
- [ ] 邀请码支持粘贴和自动查询。
- [ ] 已经是成员时直接回房间。
- [ ] 房间已满、已开始、已解散有明确页面状态。

### 23.6 等待房间

- [ ] 玩家列表实时更新。
- [ ] 角色选择由服务端锁定。
- [ ] 同一角色不能被两人选择。
- [ ] 准备状态实时更新。
- [ ] 房主权限正确。
- [ ] 房主退出必须转让或解散。

### 23.7 我的游戏

- [ ] 不依赖 localStorage 作为唯一进度来源。

## 24. 实施优先级

### 第一阶段：打通最短闭环

```text
P0-01 登录 / 注册
P0-02 剧本详情
P0-07 我的游戏
P0-08 结局页
```

目标：

```text
首页 → 剧本详情 → 登录 → 单人选角 → 主游戏 → 结局 → 我的游戏
```

### 第二阶段：打通真人多人入口

```text
P0-04 创建房间
P0-05 加入房间
P0-06 房间等待与准备
```

目标：

```text
创建 / 邀请 → 加入 → 选角色 → 准备 → 开始多人局
```

### 第三阶段：完善多人发现

```text
P0-03 房间大厅
```

公开房间可以晚于邀请制房间上线。

MVP 最低可先支持：

```text
创建私密房间
邀请链接加入
等待房间
多人开始
```

然后再开放真实公共房间列表，减少早期空大厅问题。

## 25. 最终产品体验定义

用户进入 Many Worlds 后，体验应当始终是：

```text
我先看到多个可以进入的世界
  ↓
我理解这个世界的冲突和角色
  ↓
系统记住我的身份和进入意图
  ↓
我可以独自进入，也可以邀请朋友
  ↓
每个人选择属于自己的角色
  ↓
我们确认准备后进入同一个世界
  ↓
每个人的决定都进入 AI 推演
  ↓
我的故事被别人改变，我也改变别人
  ↓
最终我得到属于自己的角色结局
  ↓
所有进度和结局都能在“我的游戏”中找到
```

最终定义：

> 首页不是房间大厅，剧本详情不是角色选择，等待房间不是聊天房，结局页也不是简单的“游戏完成”。完整产品必须让用户从发现世界、确认身份、组建房间、选择角色、进入推演，到获得个人命运结局，每一步都有明确页面承接。
