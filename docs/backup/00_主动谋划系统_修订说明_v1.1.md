# AI 多人局｜主动谋划系统修订说明 v1.1

> 目标：将“主动谋划 / 谋划机会”正式写入产品、玩法、状态机、API、配置、存储、测试、成本与试玩文档，避免开发只改 UI 而没有后端闭环。

## 一、统一术语

```text
中间：主线决策
右侧：主动谋划
次数资源：谋划机会
输入区：自拟谋划
按钮：执行谋划
```

禁止在产品文案和 UI 中使用：

```text
AP
行动点
行动力
筹谋
筹谋机会
```

代码与 API 统一使用：

```text
maneuver
maneuverType
maneuverOpportunitiesPerDay
maneuversUsedToday
maneuverOpportunitiesRemaining
```

## 二、最终规则

```text
第 1—6 天每天固定 2 次主线决策
第 1—6 天每天提供 2 次谋划机会
主动谋划可选，未使用不结转
完成两次主线决策即可进入下一天
主动谋划不能替代主线决策
第 7 天不提供主线决策和主动谋划，只进行最终裁决
```

主动谋划支持：

```text
人物交谈
派遣调查
使用筹码
自拟谋划
```

成功谋划必须进入同一消息流，并影响关系、线索、风险、任务、筹码、FateSeed 或后续事件。

## 三、为什么 10 份文档都需要修改

| 文档 | 必须修改的原因 |
|---|---|
| MVP 唯一工程基线 | 锁定术语、次数、边界和最高优先级 |
| StoryRun 状态机与 API | 增加谋划允许状态、计数、幂等、错误码和提交接口 |
| 剧本配置 Schema | 增加 maneuvers.json、leverage.json 和每日机会配置 |
| 自动化验收矩阵 | 验证次数、重置、Guard、幂等、因果和 UI |
| 部署、日志与 AI 成本 | 主动谋划最多增加 12 次推演与 12 次 Guard，必须调整预算 |
| 用户试玩方案 | 验证用户能否区分主线决策与主动谋划，以及谋划是否产生布局感 |
| Web MVP PRD | 修改主页面右栏、交互闭环、组件和接口 |
| 数据库与存储架构 | 增加 maneuverState、事件类型、事务与恢复 |
| 游戏玩法说明 | 定义两套系统的玩法差异和每日规则 |
| 《桑田诏》剧本 | 给首个剧本配置实际可用的人物、调查和筹码谋划 |

## 四、核心 API

```http
POST /api/v4/story-runs/:runId/maneuvers
```

请求核心字段：

```json
{
  "maneuverType": "contact | investigate | leverage | custom",
  "targetRoleKey": "optional",
  "intentKey": "optional",
  "leverageKey": "optional",
  "customText": "optional",
  "version": 7,
  "idempotencyKey": "uuid"
}
```

## 五、关键不变量

```text
第 1—6 天每天最多成功执行 2 次主动谋划
第 7 天不能主动谋划
被 Guard 拒绝不扣机会、不增加 version
主动谋划不改变 decisionsCompletedToday
未使用机会不结转
同一幂等键不能重复扣减
谋划结果必须有 originEventId
```
